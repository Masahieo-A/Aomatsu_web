# coding-rules.md — 実装ルール

全アプリ共通の実装ルール。アプリ固有の事情がある場合は各アプリの README に明記する。

---

## 命名規則

### フォルダ
- アプリ: `app-用途名`（すべて小文字、単語区切りはハイフン）
  - 例: `app-cloze-maker` / `app-elsa-like` / `app-eisaku-tensaku`
- 共有: `shared/components` / `shared/styles` / `shared/utils`

### ファイル
- ドキュメント: 小文字・ハイフン・`.md`（例: `design-system.md`）
- React コンポーネントファイル: `PascalCase.tsx`（例: `ClozeForm.tsx`）
- それ以外のTS/JS: `kebab-case.ts`（例: `parse-sheet.ts`）
- CSS: `kebab-case.css`

### コンポーネント
- コンポーネント名は `PascalCase`（例: `GradeSelector`）
- 1ファイル1主要コンポーネントを基本とする

### 関数・変数
- 関数・変数は `camelCase`（例: `generateClozeText`）
- 定数は `UPPER_SNAKE_CASE`（例: `MAX_BLANKS`）
- boolean は `is` / `has` / `can` プレフィックス（例: `isLoading`）

### App ID
- `APP` + 3桁連番（`APP001`〜）。一度使った番号は再利用しない。

---

## コメント方針

- **なぜそうしているか**を書く。何をしているかはコードで表現する。
- 日本語コメントOK（教育現場向けの保守性優先）。
- 自明なコメントは書かない。複雑なロジック・外部API都合・回避策にだけ付ける。
- 既存ファイルのコメント密度・文体に合わせる。

---

## 状態管理

- まず React の `useState` / `useReducer` で十分か検討する。
- グローバルな状態が必要になるまで状態管理ライブラリを足さない。
- サーバーデータは Server Components / fetch を基本にし、不要なクライアント状態を作らない。

---

## エラーハンドリング

- ユーザーには分かりやすい日本語メッセージを表示する（スタックトレースを見せない）。
- 外部API呼び出しは必ず失敗時の分岐を用意する（タイムアウト・キー未設定・レート制限）。
- 想定外エラーはログに残しつつ、UIは安全な状態に戻す。
- APP001（Gemini）など外部API利用アプリは「キー未設定」のときに明確な案内を出す。

---

## 環境変数の扱い

- 秘密情報は `.env.local`（ローカル）/ Vercel Environment Variables（本番）に置く。
- **`.env` / `.env.local` を Git にコミットしない**（`.gitignore` 済み）。
- 変数を追加したら `.env.example` に「キー名のみ（値は空 or ダミー）」を追記する。
- クライアントに露出してよい値だけ `NEXT_PUBLIC_` を付ける。秘密鍵には付けない。

---

## 依存関係

- 各アプリの `package.json` は独立。アプリ間で直接 import しない。
- 共通化したいコードは `shared/` に置く方針（導入時はオーナーに相談）。
- 依存追加・メジャーアップデートは**事前提案・承認制**。

---

## 大規模変更のルール（最重要）

- フォルダ移動 / リネーム / 一括置換 / 依存の大規模更新 / 削除は、**実行前に影響範囲を説明して承認を得る**。
- 削除は特に慎重に。対象を必ず確認し、バックアップ・退避を先に行う。
- 変更後は対象アプリで `npm run build` を通し、変更ファイルのパスを明示する。

---

## Git

- 秘密情報・`node_modules`・`.next` 等のビルド成果物はコミットしない（`.gitignore` 済み）。
- コミットは小さく、意味のある単位で。メッセージは日本語/英語どちらでも可だが内容を具体的に。
- ブランチ・push・PR はオーナーの指示があるときに行う。
