# 要件定義書: 「未来の図書館」画像生成Webアプリ

## 0. このドキュメントの目的

本ドキュメントは、Claude Codeが本アプリを実装するための要件定義書である。  
**実装前に本ドキュメントを通読し、不明点があれば実装に着手する前に質問すること。**

---

## 1. プロジェクト概要

### 1.1 背景
高校英語の授業で「未来の図書館 (The Library of the Future)」をテーマに学習を行う。学習後の発展活動として、生徒は「自分の街に欲しい図書館の1フロア」を考案し、1分間の英語プレゼンテーションを行う。  
本アプリは、その**プレゼンテーション用ビジュアルを生徒自身がAIで生成するためのツール**である。

### 1.2 アプリのゴール
- 生徒が日本語または英語のテキストで構想を入力すると、それに基づく「理想の図書館の1フロア」画像が生成される。
- 生徒は1回目の結果を見て構想を練り直し、もう1回だけ生成し直せる(計2回まで)。
- 生成画像を端末に保存できる。

### 1.3 想定ユーザー
- 学校ドメイン (例: `@example-school.ed.jp`) のGoogleアカウントを持つ高校生 約300名。
- 同時アクセスは最大1クラス分(40名前後)を想定。

## 3. 機能要件

### 3.1 認証機能

#### 3.1.1 ログイン
- トップページにアクセスすると、未ログインユーザーは**ログインページ**にリダイレクトされる。
- 「Googleでログイン」ボタンのみを設置する。
- NextAuth.jsのGoogle Providerを使用する。

#### 3.1.2 学校ドメイン制限 (重要)
- `signIn` コールバック内で、認証されたメールアドレスのドメインを検証する。
- 許可ドメインは**環境変数 `ALLOWED_EMAIL_DOMAIN`** で設定する (例: `example-school.ed.jp`)。
- 許可ドメイン以外のアカウントでログインを試みた場合は、`false` を返してログインを拒否し、ログインページに「このアプリは本校の生徒のみ利用できます」とエラーメッセージを表示する。
- セッション情報には `email` と `name` を保持する。

```ts
// 実装イメージ
callbacks: {
  async signIn({ user }) {
    const allowedDomain = process.env.ALLOWED_EMAIL_DOMAIN;
    if (!user.email?.endsWith(`@${allowedDomain}`)) return false;
    return true;
  }
}
```

#### 3.1.3 セッション戦略
- JWT セッション戦略を採用 (Supabase Adapterは使わず、利用回数管理のみDB側で行う)。
- ログアウトボタンをヘッダー右上に常時設置。

---

### 3.2 利用回数制限機能

#### 3.2.1 仕様
- **1ユーザーあたり画像生成は通算2回まで**。
- カウントは**画像生成APIがレスポンスを正常返却した時点 (= 画像が実際に生成された時点)** で +1 する。  
  → API呼び出し失敗・タイムアウト時はカウントしない (生徒救済のため)。
- 一度カウントが2に達したユーザーは、それ以降「送信」ボタンを押下できない (ボタンをdisabledにし、「利用回数の上限に達しました」と表示)。

#### 3.2.2 Supabaseテーブル設計

```sql
-- users テーブル
create table users (
  id uuid primary key default gen_random_uuid(),
  email text unique not null,
  name text,
  generation_count int not null default 0,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- generations テーブル (監査・運用ログ用)
create table generations (
  id uuid primary key default gen_random_uuid(),
  user_email text not null,
  user_prompt text not null,
  image_url text,           -- Supabase Storageに保存する場合のパス。MVPではnull可
  status text not null,     -- 'success' | 'error'
  error_message text,
  created_at timestamptz not null default now()
);

create index on generations (user_email, created_at desc);
```

- Row Level Security (RLS) は本MVPでは**有効にしない** (サーバーサイドAPIルートからService Role Keyで操作するため)。将来的に強化する場合は別途検討。

#### 3.2.3 カウント増加処理
- 画像生成成功直後に、サーバー側 (API Route) で以下を実行:
  1. `users` テーブルに該当 email のレコードがなければ INSERT (count=1)
  2. あれば `generation_count = generation_count + 1` で UPDATE
  3. `generations` テーブルに監査ログを INSERT
- これらの処理はトランザクションでなくともよいが、**生成API成功 → カウント更新の順序を守る**こと。

### 3.3 画像生成機能

#### 3.3.1 システムプロンプトとユーザープロンプトの合成

- **システムプロンプトは環境変数 `SYSTEM_PROMPT_TEMPLATE` で管理**する。  
  テンプレート内の `{{USER_PROMPT}}` プレースホルダにユーザー入力をそのまま差し込む。
- 例 (デフォルト値として `.env.example` に記載する):

```
Create a warm, photorealistic illustration of one floor of an ideal future library, viewed as a wide interior scene. The library is not only a place for borrowing books but also a community hub where people gather, learn, and create. Render the following student idea faithfully and beautifully:

---
{{USER_PROMPT}}
---

Style: bright natural lighting, friendly atmosphere, detailed but not cluttered, suitable for a high school presentation slide. No text or letters in the image.
```

- 教員が文言を変更したい場合は環境変数を書き換えるだけで対応できるようにする。

#### 3.3.2 OpenAI API呼び出し仕様

| 項目 | 値 |
|---|---|
| モデル | `gpt-image-1` |
| size | `1536x1024` (横長・プレゼン用) |
| quality | `medium` (コスト最適化) |
| n | 1 |
| response_format | base64 (`b64_json`) を受け取り、フロントへはdata URLとして返却 |

- API キーは環境変数 `OPENAI_API_KEY` から読む。**絶対にクライアント側に露出させない**こと。
- 呼び出しはすべて Next.js の Route Handler (`app/api/generate/route.ts`) 経由で行う。

#### 3.3.3 サーバー側処理フロー (`POST /api/generate`)
1. セッション検証。未ログインなら401。
2. ドメイン再検証 (二重防御)。NGなら403。
3. Supabase `users` テーブルから該当ユーザーの `generation_count` を取得。
4. **2以上なら429** (`{ error: "limit_reached" }`) を返す。
5. `SYSTEM_PROMPT_TEMPLATE` の `{{USER_PROMPT}}` をリクエストボディの `userPrompt` で置換。
6. OpenAI APIへリクエスト送信。
7. 成功時: `users.generation_count` を +1、`generations` に成功ログINSERT、画像をbase64で返却。
8. 失敗時: `generations` にエラーログINSERT、500を返す (カウントは増やさない)。

### 3.4 UI / 画面仕様

#### 3.4.1 画面構成 (シングルページ)
レイアウトは縦スクロールの1画面構成とし、**画面遷移は発生させない**。

```
┌──────────────────────────────────────┐
│  ヘッダー: アプリ名 / ユーザー名 / ログアウト │
├──────────────────────────────────────┤
│  [概要セグメント]                          │
│  - 課題の説明 (未来の図書館の1フロア)        │
│  - 利用方法 (英語または日本語でアイデアを記述) │
│  - 残り生成回数: 2 / 2                    │
├──────────────────────────────────────┤
│  [ユーザープロンプト入力枠]                  │
│  <textarea> 5〜8行、placeholder付き        │
├──────────────────────────────────────┤
│  [送信ボタン]                              │
│  (生成中はスピナー表示、ボタンdisabled)       │
├──────────────────────────────────────┤
│  [生成画像 1枚目]                          │
│  - 画像表示                                │
│  - 「端末に保存」ボタン                      │
│  - そのとき送信したプロンプト全文も小さく表示   │
├──────────────────────────────────────┤
│  [生成画像 2枚目] (2回目生成時のみ表示)       │
│  - 同上                                   │
└──────────────────────────────────────┘
```

#### 3.4.2 入力枠の挙動
- `<textarea>` で複数行入力可。
- placeholder: `例: A floor with a small café, plants everywhere, and reading nooks shaped like trees. Students can record podcasts in soundproof booths.`
- 文字数下限・上限は厳密には設けないが、500文字を超えたら警告 (送信は可能)。

#### 3.4.3 送信ボタン
- 「画像を生成する (Generate Image)」のラベル。
- 押下中はスピナー + 「生成中... (約20〜40秒かかります)」表示。
- 利用回数が上限のときは disabled + 「上限に達しました」表示。

#### 3.4.4 生成画像表示
- 生成された画像は配列で state 保持し、配列順に縦に並べる。1枚目→2枚目の順。
- 各画像の下に「端末に保存」ボタンを設置。クリックすると `<a download>` でPNGダウンロード。ファイル名は `future-library-{連番}-{yyyyMMdd-HHmmss}.png`。

#### 3.4.5 UI言語
- **UIは日本語をデフォルト**とする (生徒向け説明文・ボタン)。
- ただし入力例 (placeholder) と一部ラベルは日英併記し、英語プロンプトでの入力を促す。

#### 3.4.6 デザイントーン
- 教育用途らしい清潔感のあるデザイン。
- Tailwind + shadcn/ui の標準コンポーネントで充分。装飾は最小限。

## 4. 非機能要件

### 4.1 セキュリティ
- OpenAI APIキー、Supabase Service Role Key、NextAuthシークレットはすべて環境変数 (Vercel側で設定) で管理し、クライアントバンドルに含めない。
- `NEXT_PUBLIC_` プレフィックスはSupabaseのpublic anon keyのみに使用 (本構成では基本的に不要)。
- 画像生成APIは必ずサーバー側Route Handler経由とし、ブラウザからOpenAIへ直接通信させない。
- CSRF対策はNextAuth.js標準の挙動に任せる。

### 4.2 コスト管理
- `gpt-image-1` medium quality / 1536×1024 の単価を事前に確認し、READMEに「300名×2回=600生成時の概算コスト」を記載する。
- 万一の濫用に備え、OpenAI Platform側で**月次の使用上限 (Hard limit)** を設定することをREADMEで案内。

### 4.3 エラーハンドリング
- ネットワークエラー、APIタイムアウト (60秒)、利用上限到達、ドメイン不一致など、想定されるエラーは日本語のわかりやすい文言で表示する。
- 開発者向けの詳細エラーは `console.error` とサーバーログ (`generations.error_message`) にのみ記録。

### 4.4 アクセシビリティ
- スマートフォン (Safari iOS / Chrome Android) での操作を想定したレスポンシブ対応。
- 画像のalt属性は「生徒が生成した未来の図書館のイメージ」等で設定。
