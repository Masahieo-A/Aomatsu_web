# 弁別実験の実施ガイド（実クラス投入前の最終検証）

仕様書 `口頭試問ver2.md` §6.1〜6.2 に定められた、導入前の妥当性検証の実施手順です。
ソフトウェアの動作確認ではなく、**「このテストは本当に、自分で書いた生徒と丸投げした生徒を見分けられるのか」**を人間相手に確かめる工程です。成績評価に使い始める前に必ず一度実施してください。

## 何を確かめるのか

根幹仮説：**自分で書いた（内容を理解している）生徒は解け、AIに丸投げした生徒は解けない。**

| 確認項目 | 合格基準 | 満たさない場合のリスク |
| :--- | :--- | :--- |
| 自作群の正答率 | **80%以上** | 真面目に書いた生徒が解けない「不当な問い」を出している（偽陽性＝最も有害） |
| 自作群と丸投げ群の正答率差 | **40ポイント以上** | テストとして弁別できていない＝存在意義に関わる |

**偽陽性の抑制（自作群80%）を、弁別力（40pt差）より優先して確認してください。**

## 準備するもの

- 協力者 4〜8名（同僚の先生、または事情を説明した生徒）
- 各協力者につき英文2本：
  - **(a) 自力英文**：協力者が辞書可・AI不可で自分で書いたもの（30〜300語・3文以上）
  - **(b) 丸投げ英文**：同じお題でAI（ChatGPT等）に書かせ、**内容を読み込まずに**提出したもの
- 実験がバレないよう、(a)(b) は別の識別ラベルで投入する（例：`exp-a-01` / `exp-b-01`）

## 手順（所要：準備30分＋テスト実施15分＋採点30分）

1. **課題を2つ作成**：「弁別実験（自作群）」「弁別実験（丸投げ群）」（同じ出質モード・3問構成）
2. **英文を投入して一括生成**：それぞれの課題に (a) 群・(b) 群を投入 →「全員分を生成」
3. **レビュー・承認**：通常運用と同じく全問レビューして承認（ここは省略しない——実運用と同じ条件にするため）
4. **テスト実施**：印刷して協力者に解いてもらう。**制限時間は本番と同じ（3問10分）**。
   - (a) 群の英文は書いた本人に、(b) 群の英文は「AIに出させただけの本人」に解かせる
5. **採点**：採点キーを使って◯/×で採点し、「採点・分析」画面に入力する
6. **集計**：ダッシュボードの生徒別集計から、(a) 群平均と (b) 群平均を比較する

## 結果の読み方と対処

| 結果 | 判断 | 対処 |
| :--- | :--- | :--- |
| 自作群80%以上・差40pt以上 | ✅ 実クラス投入OK | 運用開始。教員修正率と正答率を監視 |
| 自作群が80%未満 | ⚠ 問いが不当に難しい | どの型で落ちたかを確認し、その型のプロンプトを改訂（改訂後は `npm run golden` で回帰確認）。設問の日本語の難しさ・時間不足も疑う |
| 差が40pt未満 | ⚠ 弁別できていない | 丸投げ群が正解できた問いの型を特定。時間圧系（型1・5）ばかりなら制限時間を短縮、型3（実体験）の比重を上げる（アイデア重視モードを使う）ことを検討 |

## 実施時の注意

- 結果は「AI使用の証拠」ではなく「内容理解度の測定がうまくいくか」の確認です。協力者にもその趣旨で説明してください。
- 1問単位のばらつきは正常です。**群としての平均差**で判断してください。
- 実験データは実運用データと混ざらないよう、実験用の課題は終了後に区別できる名前のまま残しておく（またはスプレッドシート上でタブごと複製して保管する）ことを推奨します。

## 実運用開始後の継続監視（ver2.md §6.3）

- 毎回の採点結果を「採点・分析」画面に入力し続けると、型別の正答率・弁別指数が蓄積されます
- ⚠フラグ（正答率>0.95／<0.2、弁別指数<0.1）が続く型はプロンプト改訂の対象です
- 改訂したら `npm run golden` → 前回レポートと比較 → 劣化がなければ次のクラスから適用、が運用サイクルです
