# 併願照合アプリ（仮称：HeiganNavi）要件定義書

- 版：v1.0（2026-07-06）
- 対象読者：Claude Code（実装担当）
- 発注者：高校進路指導担当教員
- 用途：高3三者面談（7月中旬〜）における併願校検討の根拠資料提示

---

## 1. 背景と目的

- 対象8大学（愛知・愛知学院・中京・愛知淑徳・金城学院・名古屋学院・南山・名城）の入試要項PDFを元データとし、**第一志望と併願候補の「科目充足」「試験日被り」「入学金先払い」を機械照合**して面談中に即提示する。
- 設計思想：**生成ではなく照合**。面談画面ではLLMを一切呼ばない。構造化データは事前にLLMで抽出するが、**逐語引用の原文一致検証＋教員確定**を経たものだけを使う。
- 日付・科目の誤りは進路事故に直結するため、全表示項目に「原文引用＋PDF該当ページへのリンク＋年度バッジ」を付け、1クリックで一次資料に到達できること。

## 2. システム全体構成（3コンポーネント）

```
[A] 抽出パイプライン（CLI / ローカル実行）
    PDF逐語テキスト(.txt) → Gemini構造化 → 原文一致検証 → 候補JSON

[B] 教員確定ツール（Google Sheets）
    候補JSONをSheetsに流し込み → 教員が確認・修正 → status=confirmed

[C] 面談アプリ（Next.js / Vercel）
    確定データのみ読込 → 検索・照合・表示（LLM呼び出しなし）
```

## 3. 元データ（Cowork収集物）

Coworkが以下の構造で収集済みの前提。パイプラインはこの構造を入力とする。

```
juken-data/
  manifest.csv                # 大学名,年度,方式区分,ファイルパス,取得元URL,取得日,公開状況メモ
  {大学名}/
    2027/       *.pdf + *.txt（=== PAGE N === 区切りの逐語テキスト）
    2026_実績/  *.pdf + *.txt
```

- `.txt` が**唯一の「正」テキスト**。検証はすべてこのファイルに対して行う。
- 2027年度版が未公開の方式は 2026_実績 のデータを `carryover` として扱う（§4）。
- juken-data/ は Google Drive で同期し、PDF本体は後述の Storage にもコピーする。

## 4. データスキーマ

### 4.1 入試方式レコード（中核テーブル：`exam_methods`）

```typescript
type ExamMethod = {
  id: string;                 // 例 "nanzan-2027-ippan-zengaku-kobetsu"
  university: string;         // "南山大学"
  faculty: string;            // "外国語学部"
  department: string;         // "英米学科"（学部一括募集は "-"）
  methodName: string;         // "全学統一入試（個別学力試験型）"
  methodCategory: "shiteiko" | "kobo" | "sogo" | "ippan" | "kyotsu";
  heiganPolicy: "senga n" | "heigan_ok" | "unknown";  // 専願/併願可/要確認
  subjects: SubjectRequirement;          // §4.2
  schedule: {
    applicationStart: string | null;     // ISO日付
    applicationEnd: string | null;
    examDates: string[];                 // 試験日（自由選択制は複数列挙）
    examDateSelectable: boolean;         // 複数日から選択可か
    resultDate: string | null;           // 合格発表日
    enrollDeadline: string | null;       // 入学手続（入学金）締切
  };
  fee: number | null;                    // 検定料（円）
  feeNote: string | null;                // 学内併願割引等は自動計算せず注記
  dataYear: 2027 | 2026;
  dataStatus: "confirmed_2027" | "carryover_2026";  // 年度バッジの根拠
  evidence: Evidence[];                  // §4.3 全数値フィールドに必須
  reviewStatus: "auto" | "flagged" | "confirmed";   // confirmedのみ面談アプリに出す
  reviewedBy: string | null;
  reviewedAt: string | null;
};
```

### 4.2 科目要件の正規化

```typescript
type SubjectRequirement = {
  required: SubjectCode[];               // 必須科目
  choiceGroups: { from: SubjectCode[]; pick: number }[];  // 「〜から1」等
  totalCount: number;                    // 表示用の教科数
  nonAcademic: ("小論文" | "面接" | "書類" | "プレゼン" | "実技" | "基礎学力調査")[];
};

type SubjectCode = {
  subject: "英語" | "国語" | "数学" | "日本史" | "世界史" | "地理" |
           "公民" | "物理" | "化学" | "生物" | "情報";
  range?: string;   // 範囲限定。例 国語:"古文漢文除く" / 数学:"ⅠAⅡBC" / "ⅠAのみ"
};
```

- 照合は **教科レベルで一次判定、range不一致は△に降格**する2段階（§8.1）。
- 「地歴公民から1」のような教科横断選択は choiceGroups で表現する。

### 4.3 根拠（Evidence）— ハルシネーション防御の中核

```typescript
type Evidence = {
  field: string;        // 例 "schedule.examDates"
  quote: string;        // 要項からの逐語引用（LLM出力）
  pdfFile: string;      // juken-data相対パス
  page: number;
  verified: boolean;    // 自動検証結果（§5.3）
};
```

## 5. 抽出パイプライン（コンポーネントA）

### 5.1 実行形態

- Node.js（TypeScript）製CLI。`npx tsx pipeline.ts --univ 南山大学` のように大学単位で実行。
- **ハッシュキャッシュ**：PDFのsha256をキーに抽出結果をキャッシュし、再実行時は未変更PDFをスキップ（ZPDアプリと同方式）。

### 5.2 LLM構造化

- モデル：Gemini 2.5 Flash（implicit caching前提。プロンプトの固定部を先頭に置く）。
- プロンプトはコード外の `prompts/extract.md` に置く（外部テンプレート方式）。
- 入力：`.txt` の全文（長大な場合は方式ごとの章単位で分割投入）。
- 出力：ExamMethod[]（JSONのみ。スキーマはプロンプト内にJSON Schemaで明示）。
- **必須指示**：「すべての数値・日付・科目フィールドについて、根拠となる原文を一字一句そのまま quote に引用し、ページ番号を付けること。原文に無い情報を補完してはならない。不明は null。」

### 5.3 原文一致検証（機械・決定的）

```
verify(quote, txtFile, page):
  normalize = 全角→半角統一, 空白・改行除去, 波ダッシュ/長音統一
  return normalize(quote) が normalize(該当ページ±1ページのテキスト) に部分一致するか
```

- 不一致 → そのフィールドを null 化し `reviewStatus = "flagged"`。
- quote が存在しても**値とquoteの整合**（日付文字列がquoteに含まれるか等）を正規表現で二次チェック。
- 検証を通らないデータが面談アプリに出る経路は存在しないこと（テストで担保）。

### 5.4 出力

- `output/candidates_{大学名}.json` を生成し、Sheets同期スクリプトで確定シートに流し込む。

## 6. 教員確定ツール（コンポーネントB）

- Google Sheets 1ブック。シート＝大学。1行＝1 ExamMethod（学部学科×方式）。
- 列は §4.1 をフラット化。evidence は「quote / page / PDFリンク」列として横持ち。
- 条件付き書式：`flagged`=赤、`carryover_2026`=黄、`confirmed`=緑。
- 教員の操作は「セル修正 → reviewStatus を confirmed に変更」のみ。
- 面談アプリは **confirmed 行だけ** を取得（Sheets API、ビルド時+ISR 1時間）。

## 7. 面談アプリ 機能要件（コンポーネントC）

### F1. 第一志望ビュー
- 大学→学部→学科をセレクトで選択（フリー入力不要。8大学固定）。
- 当該学科の全方式を時系列カードで表示：方式名 / 方式区分バッジ / 併願可否バッジ / 科目 / 出願期間 / 試験日 / 発表日 / 手続締切 / 検定料 / **年度バッジ（2027確定 or 2026実績=黄）** / 各項目の「根拠」ボタン。
- 「根拠」ボタン → 原文引用をポップオーバー表示＋「PDFで確認」リンク。

### F2. 基準方式の指定
- F1のカードから1つを「基準方式」として選択（例：一般・3教科型）。
- 基準方式の科目集合が画面上部に固定表示される（例：英・国・日本史）。

### F3. 併願候補の科目照合
- 併願候補の大学（複数可）を選ぶと、その全方式を判定結果つきで一覧表示：
  - **○ 完全充足**：基準の科目集合のままで受験可
  - **△ 条件付き**：(a) 追加1科目で受験可（不足科目名を明示）、(b) 教科は一致するが range 不一致（例：国語の古文有無）
  - **× 不可**：追加2科目以上が必要
- 小論文・面接等 nonAcademic のみの方式は「学科試験なし（書類・面接型）」として別枠表示。

### F4. 日程照合
- 選択中の方式群（基準＋併願候補）を横軸カレンダー（10月〜3月）で表示。
- 被り判定：
  - 双方 confirmed_2027 → 同日重複を赤表示
  - どちらかが carryover_2026 → **±3日を黄色帯**にし「前年度実績・要確認」表示
  - examDateSelectable=true の方式は選択可能日の集合で判定し「回避可能」注記
- 連続受験3日超の並びに警告アイコン。

### F5. 入学金・費用ビュー
- 各併願方式について `手続締切 < 基準方式の発表日` なら「**入学金先払い発生**」バッジ。
- 選択方式の検定料を単純合算表示（割引は feeNote をそのまま表示、自動計算しない）。

### F6. 面談サマリー出力
- 選択結果（基準＋併願一覧＋照合結果＋費用）を印刷用1枚レイアウト（A4）で表示 → ブラウザ印刷。既存の「併願校決定シート」STEP5にそのまま転記できる列構成にする。

### 全画面共通
- フッター常設：「本画面は要項の抜粋です。出願前に必ず要項原本を確認してください」＋データ最終更新日。
- 生徒の個人情報は**入力させない・保存しない**（照合は大学・方式の選択のみで完結）。

## 8. 照合アルゴリズム

### 8.1 科目充足判定

```typescript
// S: 基準方式から導いた保有科目集合（教科レベル）
function judge(S: Set<Subject>, m: ExamMethod): "ok" | "plus_one" | "ng" {
  const need = (extra: Subject | null) => {
    const have = extra ? new Set([...S, extra]) : S;
    if (!m.subjects.required.every(r => have.has(r.subject))) return false;
    // choiceGroups: 貪欲でなくマッチング（同一科目の二重使用禁止）で充足判定
    return canSatisfyChoiceGroups(m.subjects.choiceGroups, have, m.subjects.required);
  };
  if (need(null)) return rangeCheck(S, m) ? "ok" : "plus_one"; // range不一致は△
  for (const s of ALL_SUBJECTS) if (!S.has(s) && need(s)) return "plus_one";
  return "ng";
}
```

- choiceGroups の充足は二部マッチング（科目数が小さいので全探索で可）。
- `plus_one` の場合、**どの科目を追加すれば充足するか**を必ず返す（UI表示用）。

### 8.2 日程被り判定

```typescript
type DateBand = { dates: string[]; fuzzyDays: 0 | 3 }; // carryoverは3
// 被り = ある試験日ペアの差 <= max(fuzzy) 。selectable方式は全日被りのときのみ「不可避」。
```

## 9. 非機能要件

- 面談中の全操作はローカルデータで完結（初回ロード後はネットワーク不要）。応答100ms以内。
- 認証：Google OAuth（学校ドメイン制限）。教員のみ利用想定。
- PDF本体は Supabase Storage に配置し、`/viewer?file=...&page=N` ルートで pdf.js により該当ページを直接開く（Driveプレビューはページ指定不可のため使わない）。
- 要項PDFは各大学の公開資料だが、**アプリは校内限定公開**とし再配布はしない。
- 個人情報を扱わないため、既存のAI利用ガイドライン上は「公開情報の整理ツール」として位置づけ可能。

## 11. 開発フェーズ

1. **P1（面談に間に合わせるMVP）**：スキーマ確定 → Sheetsに**主要学科×主要方式を手入力**（教員作業、1大学30分想定）→ F1〜F4実装。抽出パイプラインなしでも稼働する。
2. **P2**：抽出パイプライン（§5）＋検証＋Sheets流し込み。手入力分と突合してパイプライン精度を実測。
3. **P3**：F5・F6、pdf.jsビューア、carryover→2027確定への差し替え運用（要項公開の秋に向けて）。

## 12. リスクと対策

| リスク | 対策 |
|---|---|
| LLM抽出の誤り | 原文一致検証＋flagged隔離＋教員確定（confirmed以外は表示不可） |
| 前年度データの日付ズレ | dataStatusバッジ＋±3日黄色帯＋「要確認」文言を判定結果に必ず併記 |
| 要項の年度途中改訂 | manifest再収集（Cowork再実行）→ ハッシュ差分のみ再抽出 |
| 表の抽出崩れ | .txt はページ単位で保持し、検証は±1ページ幅で照合。崩れが激しい方式は手入力へフォールバック |
| 選択科目表現の解釈ミス | choiceGroups化はflagged既定とし、初年度は全件教員確認 |

## 付録A：抽出プロンプト（prompts/extract.md）の骨子

1. 役割：入試要項からの構造化抽出専用。創作・推測・補完の禁止を明記。
2. 出力：ExamMethod[] の JSON のみ（スキーマ全文を貼付）。
3. 引用規則：数値・日付・科目の各フィールドに evidence（逐語quote＋page）必須。quoteは原文の連続する文字列であること。
4. few-shot：南山（全学統一・選択科目あり）、名城（公募・基礎学力調査型）、共テ利用（学科試験なし）の3例を実データから作成する。
5. 不明・記載なしは null。「前年同様」等の記述はそのまま quote し値は null。
