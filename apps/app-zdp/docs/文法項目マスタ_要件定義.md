# 文法項目マスタ_要件定義.md
# 文法項目マスタ管理Webアプリ（開発者個人用）要件定義書

> **位置づけ**: 本アプリは ZDPアプリ（`ZDP_要件定義.md`）の「脳」となる知識資産 `grammar_master.json` を作成・検証・育成するための、**開発者（英語教員）個人向けツール**である。ZDPアプリの診断精度はこのマスタの品質で決まるため、UIの華美さより「知識を書きやすく、書いた知識の効果を即座に検証できる」ことを最優先する。

---

## 1. 設計思想

### 1.1 このアプリが守るべき3原則

1. **知識層とアプリ層の分離**
   マスタは純粋なJSONデータ資産として管理する。将来LLMのモデルが世代交代しても、ZDPアプリを作り直しても、この資産はそのまま持ち越せる。
2. **テストベンチ中心主義**
   「項目を書く → Flash に判定させる → 精度を見る → 記述を直す」のループを最短化する。マスタの各項目は**書いた時点では仮説**であり、Flash が正しく判定できて初めて資産になる。250項目書いてから精度が出ないと判明するのが最悪のシナリオ。**1項目ずつ検証しながら書く**ためのツールである。
3. **授業起点の逐次拡充**
   項目は一気に網羅しない。授業・添削で実際に出会った生徒の躓きから項目を起こす運用を前提に、「メモ→下書き→検証→公開」のステータス管理を持つ。

### 1.2 マスタが表現する4つの知識

| 知識 | 誰の知見か | ZDPアプリでの役割 |
|------|-----------|------------------|
| 文法項目の判定基準・陽性例・陰性例 | 教員 | P-01 タグ付けの few-shot |
| **前提項目グラフ**（この項目の理解に必要な項目） | 教員 | 診断の遡り順序 = ZPDの操作的定義 |
| **頻出誤解パターン** | 教員（誤答を見てきた経験） | プローブの誤答選択肢・フィードバック |
| プローブ雛形 | 教員 | P-02 切り分け質問の生成制約 |

この4つは市販の文法書には載っていない。**教室で誤答を見てきた経験をデータ化する**のが本アプリの本質。

## 2. データスキーマ（grammar_master.json）

ZDPアプリとの共有スキーマ。**変更時は両アプリで同期すること。**

```json
{
  "schema_version": "1.0",
  "exported_at": "2026-07-04T00:00:00+09:00",
  "items": [
    {
      "id": "G045",
      "name": "分詞構文（過去分詞・挿入型）",
      "name_en": "participial construction (past participle, parenthetical)",
      "status": "published",
      "level": "B1",
      "parent_id": "G040",
      "prerequisites": ["G031", "G012"],
      "criteria": "コンマで区切られた過去分詞句が主節の主語を修飾している",
      "patterns": [", first proposed in ...,", ", written by ...,"],
      "positive_examples": [
        {
          "sentence": "The theory, first proposed in 1984, has since been challenged.",
          "span": "first proposed in 1984"
        }
      ],
      "negative_examples": [
        {
          "sentence": "The proposed theory has been challenged.",
          "reason": "過去分詞の前置修飾であり、挿入型ではない（G031に該当）"
        }
      ],
      "common_misconceptions": [
        "挿入句内の過去分詞を主節の動詞と誤読する",
        "proposed を過去形の動詞と解釈し、文に動詞が2つあると混乱する"
      ],
      "probe_template": "挿入句を除去した文を提示して主節の理解を確認する。次に挿入句のみを取り出し、分詞の意味上の主語を問う。",
      "notes": "2026年度3年A組で頻出。教科書 Lesson 4 に出現。",
      "test_results": {
        "last_tested_at": "2026-07-01T00:00:00+09:00",
        "accuracy": 0.95,
        "runs": 20
      },
      "created_at": "...",
      "updated_at": "...",
      "version": 3
    }
  ]
}
```

### スキーマ上の重要な規約
- `id`: `G` + 3桁連番。**一度発行したIDは削除・再利用しない**（ZDPアプリ側の weakness_history が参照するため）。廃止する場合は `status: "deprecated"` にする
- `status`: `memo`（授業メモ段階）→ `draft`（記述完了）→ `verified`（テストベンチ通過）→ `published`（ZDPアプリに供給可）→ `deprecated`
- `prerequisites`: **有向非巡回グラフ（DAG）を構成すること。循環参照はアプリが保存時に拒否する**
- `positive_examples` は3件以上、`negative_examples` は2件以上を `verified` への昇格条件とする
- `level`: CEFR（A1〜C1）。学習指導要領との対応は `notes` に記載

## 3. 機能要件

### 3.1 項目エディタ（中核機能①）
| ID | 機能 |
|----|------|
| E-01 | 項目のCRUD。スキーマ準拠のフォーム（必須項目のバリデーション付き） |
| E-02 | クイックメモ機能: スマホから「今日の授業で○○の誤読が多発」程度のメモを最速で登録（status: memo）。後日PCで清書 |
| E-03 | 陽性例・陰性例の追加UI。英文を貼ると該当spanをドラッグ選択で指定できる |
| E-04 | 既存項目の複製から新項目を作成（類似項目の記述を効率化） |
| E-05 | 変更履歴の保持（version インクリメントと変更前JSONの保存） |

### 3.2 テストベンチ（中核機能②・最重要）
| ID | 機能 |
|----|------|
| B-01 | 選択した項目について、登録済みの陽性例・陰性例を Gemini Flash（P-01と同一プロンプト）に判定させ、正答率を表示する |
| B-02 | self-consistency（3回多数決）の有無を切り替えて比較できる |
| B-03 | 誤判定された例文と、その際のLLM出力（evidence含む）を一覧表示。**どの記述（判定基準？陽性例？）を直せば改善するかの手がかりにする** |
| B-04 | テスト用の追加例文（マスタに保存しない一時例文）での判定試行 |
| B-05 | 精度基準: 正答率90%以上・runs 10回以上で `verified` に昇格可能。テスト結果は `test_results` に記録 |
| B-06 | 全published項目の一括回帰テスト（プロンプト改訂時・モデル更新時に実行） |

> **B-06 の意義**: Gemini Flash のモデル更新やプロンプト改訂で判定精度が変わる可能性がある。マスタを「テストスイート付きの知識ベース」として維持することで、モデル世代交代に強い資産になる。

### 3.3 前提項目グラフ（中核機能③）
| ID | 機能 |
|----|------|
| G-01 | prerequisites に基づく依存グラフの可視化（Mermaid または D3。ズーム・項目クリックでエディタへ遷移） |
| G-02 | 循環参照の検出と保存拒否 |
| G-03 | 「孤立項目」（どこからも参照されず、前提も持たない項目）の一覧表示 → グラフの穴の発見 |
| G-04 | ある項目を選ぶと「そこに至る学習経路」（前提の連鎖）をハイライト表示 |

### 3.4 入出力
| ID | 機能 |
|----|------|
| IO-01 | `grammar_master.json` のエクスポート（published のみ / 全件 を選択可） |
| IO-02 | JSONインポート（バックアップ復元・他環境からの取り込み） |
| IO-03 | Googleスプレッドシートへの同期（閲覧・バックアップ用。編集はアプリ側を正とする一方向同期） |
| IO-04 | 項目一覧のCSVエクスポート（研究発表・実践報告の資料用） |

### 3.5 ダッシュボード
| ID | 機能 |
|----|------|
| D-01 | ステータス別項目数（memo / draft / verified / published） |
| D-02 | レベル別・カテゴリ別のカバレッジ表示（学習指導要領の文法事項リストと突き合わせ、未作成領域を可視化） |
| D-03 | ZDPアプリからの `unclassified` ログの取り込み表示（Phase 2。マスタ拡充の需要データ） |

## 4. 技術スタック

| 層 | 技術 | 備考 |
|----|------|------|
| フロント/バック | Next.js（App Router） | ZDPアプリと技術を揃え、コード知見を共有 |
| データ保存 | **正（マスターデータ）はGoogle Drive上の作業用JSON（`master_working.json`・全ステータス含む）とする**。保存のたびにタイムスタンプ付きバックアップを同フォルダに自動生成。スプレッドシート同期（IO-03）は閲覧用の従属コピー | 個人用ツールなのでDBは最小構成。「正」が1箇所であることを厳守 |
| LLM | Gemini API（gemini-flash 系統） | **ZDPアプリの P-01 プロンプトと完全に同一のものを使用すること**（テストベンチの意味を担保） |
| グラフ描画 | Mermaid.js（簡易）→ 必要なら D3 | |
| 認証 | 開発者個人のみ（Basic認証程度で可） | |
| ホスティング | Vercel またはローカル実行 | ローカル実行でも成立する設計にする |

> **重要**: テストベンチのプロンプトは ZDPアプリの `ZDP_プロンプト設計.md` P-01 を共通パッケージ化して両アプリから参照する（コピーの分岐を防ぐ）。モノレポ構成（`apps/zdp`, `apps/master-editor`, `packages/prompts`, `packages/schema`）を推奨。

## 5. 運用フロー（想定される日常）

```
【授業中・添削中】
 生徒の躓きに気づく
   → スマホでクイックメモ登録（30秒。status: memo）

【放課後・週末】
 memo を開いて清書
   → 判定基準・陽性例3・陰性例2・誤解パターン・プローブ雛形を記述（status: draft）
   → テストベンチ実行
       精度 < 90% → B-03 で誤判定を分析 → 判定基準・例文を修正 → 再テスト
       精度 ≥ 90% → verified に昇格
   → 前提グラフで位置づけを確認・prerequisites を設定
   → published に昇格

【月1回程度】
 published 全件をエクスポート（第6章の引き継ぎ仕様に従う） → ZDPアプリへ供給
 一括回帰テスト（B-06）を実行して健全性確認
```

## 6. ZDPアプリへのデータ引き継ぎ仕様（本アプリの出口・最重要インターフェース）

本アプリの存在意義は、品質が担保された知識データをZDPアプリに供給することにある。引き継ぎは以下の仕様に厳密に従う。

### 6.1 引き継がれるデータの定義

| 観点 | 仕様 |
|------|------|
| **形式** | `grammar_master.json`（第2章のスキーマに準拠、UTF-8） |
| **対象** | `status: "published"` の項目**のみ**。memo / draft / verified は含めない |
| **品質保証の根拠** | published は「陽性例3件以上・陰性例2件以上・テストベンチ正答率90%以上（runs 10以上）」を通過した項目のみが到達できるステータスであるため、このファイルに含まれる全項目は検証済みであることが構造的に保証される |
| **メタ情報** | `schema_version` / `exported_at` / 総項目数 を必ず含める |

### 6.2 エクスポート時の自動検証（1つでも失敗したらエクスポート中止）

| # | 検証 | 失敗時 |
|---|------|--------|
| V-1 | スキーマ準拠チェック（必須フィールド・型） | 該当項目を列挙して中止 |
| V-2 | **参照整合性: published項目の `prerequisites` および `parent_id` が指す先も必ずpublishedであること** | 「G045の前提G031が未publishedです」のように列挙して中止。※これを許すとZDPアプリの前提グラフ遡り（診断の核心）が参照切れで壊れる |
| V-3 | DAG検証（published集合内で循環がないこと） | 循環パスを表示して中止 |
| V-4 | ID重複チェック | 中止 |

### 6.3 保存場所と受け渡し経路

```
[本アプリの正データ]
 Google Drive: /文法項目マスタ/master_working.json（全ステータス・編集用）
        │
        │ エクスポート実行（6.2の検証通過時のみ）
        ▼
[引き継ぎ成果物]
 Google Drive: /文法項目マスタ/exports/grammar_master_v{n}_{日付}.json
  ├─ エクスポートごとに連番・日付付きで保存（上書きしない）
  └─ 最新版のコピーを grammar_master_latest.json として同フォルダに配置
        │
        │ MVP: ZDPアプリの管理画面（T-02）から手動アップロード
        │ Phase 2: ZDPアプリが起動時にDriveの latest を直接読む（Drive API・読み取り専用）
        ▼
[ZDPアプリ]
 読み込み時に schema_version を照合。
 非対応バージョンなら読み込みを拒否し、直前の正常版で稼働を継続する
```

### 6.4 バージョン互換の規約

- `schema_version` は「メジャー.マイナー」形式。**フィールド追加はマイナー更新（ZDPアプリは未知フィールドを無視して読める設計にする）、フィールドの削除・意味変更はメジャー更新**とし、メジャー更新時は両アプリを同時改修する
- ZDPアプリ側の `weakness_history` は項目IDで蓄積されているため、**IDの永続性（削除・再利用禁止、廃止は deprecated）が引き継ぎ契約の一部**である
- deprecated になった項目はエクスポートに含めないが、ZDPアプリは過去履歴中の未知IDを「アーカイブ項目」として表示だけできるよう、エクスポートに `deprecated_ids: []` の一覧を付す

## 7. 初期データ計画

- **最初のスプリント: 20項目**（スキーマ妥当性の検証が目的。作り込みすぎない）
  - 選定基準: ①担当クラスの直近の誤答で頻度が高い ②教科書の現行レッスンに登場する ③前提関係が明確でグラフの背骨になる（例: 受動態 → 過去分詞の形容詞用法 → 分詞構文）
- 20項目でテストベンチの精度傾向を確認し、**スキーマ改訂が必要ならこの段階で行う**（スキーマ変更は後になるほど高コスト）
- 中期目標: 150〜250項目（高校英語の主要文法事項をカバー）

## 8. 非機能要件・留意事項

- **データ喪失防止**: 保存のたびにタイムスタンプ付きバックアップJSONを自動生成（この資産の喪失が最大のリスク）
- **API コスト**: テストベンチは開発者1名の利用なので問題にならない見込みだが、B-06 一括回帰は published 数 × 例文数 × 3回の呼び出しになるため、実行前に概算呼び出し数を表示する
- **知的資産としての管理**: マスタJSONは実践研究（CELES等）の中核データになり得る。エクスポート時に CC ライセンス表記等を付与できるフィールドを schema_version 1.1 で検討
- **書きやすさ最優先**: フォームの入力項目が多いため、Gemini Flash による「下書き支援」（例文から判定基準のドラフトを生成 → 教員が修正）を E-06 として Phase 2 で検討。ただし**教員の修正を必須とする**（LLM生成のまま保存させない）

## 9. 受け入れ基準（MVP）

1. スキーマ準拠の項目を作成・編集・保存でき、循環参照が拒否される
2. 1項目についてテストベンチを実行し、正答率と誤判定詳細が表示される
3. 依存グラフが可視化され、項目クリックでエディタに遷移できる
4. published 項目のみを `grammar_master.json` としてエクスポートでき、そのファイルが ZDPアプリでそのまま読み込める
   - 6.2 の検証（参照整合性・DAG・スキーマ）が実装され、未published前提を持つ項目のエクスポートが正しく拒否されること
5. スマホからクイックメモ登録ができる
