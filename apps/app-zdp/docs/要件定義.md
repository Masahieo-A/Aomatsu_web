# ZPD診断型 英語学習支援Webアプリ 要件定義書

> **Claude Code への指示**: 本ドキュメントを唯一の正とする。プロンプト本文は同梱の `ZDP_プロンプト設計.md` に定義されており、実装時はそのファイルからプロンプトを読み込む構造にすること（ハードコード禁止）。文法項目マスタは `grammar_master.json` として外部ファイル化し、差し替え可能にすること。

## 1. プロジェクト概要

### 1.1 目的
生徒が「教科書のこの英文が分からない」と入力したとき、理解を妨げている原因（語彙／熟語／文型／文法項目／談話構造）を**切り分け診断**し、原因に応じた類似例文で反復学習させるWebアプリを構築する。

理論的基盤:
- **ヴィゴツキーのZPD（発達の最近接領域）**: 「一人でできる水準」と「支援があればできる水準」の差に働きかける
- **ブルームの2シグマ**: 1対1個別指導の効果をAIでスケールする
- **Krashen の i+1**: 出題難易度は現水準+1段階に制御する
- **Corder の誤り分析**: Mistake（偶発的ミス）と Error（体系的未習得）を区別し、Errorのみ介入する

### 1.2 制約条件
- **使用API: Gemini Flash系統のみ**（gemini-flash-latest 相当）。上位モデルは使用しない
- 弱いモデルで精度を出すため、後述の「ブレ最小化アーキテクチャ」を必須とする
- データベースは **Google スプレッドシート** で管理する（学校のGoogle Workspace環境を想定）
- 開発初期は文法項目マスタが最小構成（ダミー20項目程度）である。マスタは別アプリ（文法項目マスタ管理ツール）で拡充され、JSONとして本アプリに供給される。**マスタの中身に依存しない設計とすること**

### 1.3 ユーザー
- 生徒（高校生）: 診断・学習を行う
- 教師（管理者）: 学習履歴の閲覧、マスタJSONの更新

## 2. システム全体フロー

```
[生徒] 分からない英文を入力（テキスト or 教科書写真OCR ※OCRはPhase 3）
   ↓
[STEP 1: 前処理（プログラム側・LLM不使用）]
   - 語彙難易度判定: CEFR-J Wordlist と照合
   - 表層パターン検出: 正規表現/品詞解析で文法項目候補を絞り込み
     （例: 「, + 過去分詞」→ 分詞構文系の項目候補）
   ↓
[STEP 2: 文法項目タグ付け（LLM・分類タスク）]
   - 絞り込まれた候補項目（5〜10件）についてのみ、
     マスタの判定基準・陽性例・陰性例を few-shot 注入し YES/NO 判定
   - 重要判定は self-consistency（3回多数決）
   ↓
[STEP 3: 診断プローブ提示]
   - 語彙プローブ: 難単語を単体で提示し意味確認
   - 構造プローブ: マスタの「プローブ雛形」に基づき生成
     （例: 挿入句を除去した文、同一構造×易しい語彙の文）
   - 各回答で「正誤 + 確信度（1〜5）」を取得
   ↓
[STEP 4: 原因特定]
   - 4象限分類: 正誤 × 確信度 → 習得済み / ZPD圏内 / Error候補 / Mistake候補
   - 同一項目で2回以上の安定した誤り → Error 確定
   - 前提項目グラフを遡り、「どこまで戻れば足場が届くか」を特定
   ↓
[STEP 5: 類似例文出題（i+1制御）]
   - Error項目 × 現水準+1難易度 で類似例文を生成
   - 生成文は検証プロンプトで二重チェック（対象項目を含むか／難易度は適正か）
   - 検証済み例文はキャッシュに保存（資産化）
   - フィードバックは「問い返し型」（答えを直接与えない）
   - 3問連続不正解で難易度を1段階下げる（自動フロア）
   ↓
[STEP 6: 産出タスク]
   - 選択 → 変形 → 自由産出 の3層で「使える」状態まで確認
   ↓
[学習者モデルDBに履歴を正規化して蓄積]
```

### 2.1 感情チェック（横断機能）
5問ごとに 😵むずかしすぎ / 😐ちょうどいい / 😌かんたんすぎ の3択を提示。「むずかしすぎ」連続で難易度低下、「かんたんすぎ」連続で再診断。

## 4. ブレ最小化アーキテクチャ（最重要・必須遵守）

Gemini Flash 系で診断精度を出すための設計原則。実装時に逸脱しないこと。

1. **生成タスクを分類タスクに変換する**
   「この英文の文法的難所を分析せよ」のような自由記述は禁止。必ず「候補項目リストから該当するものをYES/NOで判定」の形にする。
2. **1プロンプト1判定**
   「分析して診断して出題して」を1回のAPI呼び出しに詰め込まない。小さく閉じたタスクに分解する。Flash は単価が安いので呼び出し回数側にコストを振る。
3. **候補絞り込みはプログラムで先に行う**
   全マスタ項目からLLMに選ばせない。正規表現・品詞タグ付け（compromise.js もしくはサーバ側 spaCy）で候補を5〜10件に絞ってから判定させる。
4. **few-shot はマスタから動的注入**
   各項目の「判定基準」「陽性例」「陰性例」をプロンプトに埋め込む。プロンプトテンプレートは `ZDP_プロンプト設計.md` 参照。
5. **構造化出力の強制**
   すべてのLLM応答は JSON スキーマを指定し、パース失敗時は1回だけリトライ。temperature は判定系 0.1、生成系 0.7。
6. **多数決（self-consistency）**
   STEP 2 のタグ付けは temperature 0.7 で3回実行し多数決。コスト目安: Flash 3回 << 上位モデル1回。
7. **キャッシュ最優先**
   教科書英文は全国共通。分析済みの文はハッシュ照合でキャッシュヒットさせ、API呼び出しをゼロにする。**運用が進むほど検証済みタグ付きコーパスが自動蓄積される**構造を意識する。

## 5. データ設計（Googleスプレッドシート）

1つのスプレッドシートに以下のシートを持つ。アクセスはサービスアカウント経由（Google Sheets API v4）。

### 5.1 シート: `learners`（学習者）
| 列 | 型 | 説明 |
|----|-----|------|
| learner_id | string | UUID |
| display_name | string | 表示名（個人情報最小化: 出席番号等） |
| current_level | string | 推定CEFRレベル（A1〜B2） |
| created_at | ISO8601 | |

### 5.2 シート: `sessions`（診断セッション）
| 列 | 型 | 説明 |
|----|-----|------|
| session_id | string | UUID |
| learner_id | string | FK |
| input_sentence | string | 入力された英文 |
| sentence_hash | string | SHA-256（キャッシュキー） |
| identified_items | string | 特定された文法項目ID（カンマ区切り） |
| root_cause | string | 最終診断（vocab / idiom / structure / grammar:{item_id} / discourse） |
| started_at / ended_at | ISO8601 | |

### 5.3 シート: `responses`（回答ログ）
| 列 | 型 | 説明 |
|----|-----|------|
| response_id | string | UUID |
| session_id | string | FK |
| probe_type | string | vocab / structure / similar / production |
| item_id | string | 対象文法項目ID（該当時） |
| question | string | 提示した問題 |
| answer | string | 生徒の回答 |
| is_correct | boolean | |
| confidence | int | 1〜5 |
| quadrant | string | mastered / zpd / error_candidate / mistake_candidate |
| answered_at | ISO8601 | |

### 5.4 シート: `weakness_history`（弱点履歴＝学習者モデルの中核）
| 列 | 型 | 説明 |
|----|-----|------|
| learner_id | string | FK |
| item_id | string | 文法項目マスタのID（統制語彙に正規化済み） |
| status | string | error / improving / mastered |
| error_count | int | 安定誤り回数 |
| last_seen | ISO8601 | |

### 5.5 シート: `sentence_cache`（分析・例文キャッシュ）
| 列 | 型 | 説明 |

| sentence | string | 原文 |
| analysis_json | string | STEP 1-2 の分析結果（JSON文字列） |

### 5.6 シート: `api_log`（コスト監視）
| 列 | 型 |
|----|-----|
| date, call_type, token_in, token_out, cached(bool) | |

> **注意**: スプレッドシートは同時書き込みに弱い。書き込みはサーバ側でキューイングし、appendRow ベースで運用する。読み込み頻度の高い `sentence_cache` と `grammar_master` はサーバメモリにキャッシュし、5分TTLで再読込。将来的に生徒数が増えた場合は Supabase 等への移行パスを想定し、データアクセス層を抽象化（Repository パターン）しておくこと。

## 6. 文法項目マスタ（外部依存）

- ファイル: `grammar_master.json`（別アプリで作成・管理される。スキーマは `文法項目マスタ_要件定義.md` に準拠）
- 本アプリは**読み取り専用**。起動時にロードし、T-02 で差し替え可能にする
- MVP開発用に、スキーマに準拠したダミー20項目（`grammar_master.sample.json`）を同梱すること。項目例: 仮定法過去、分詞構文（挿入型）、関係代名詞の省略、時制の一致、受動態、to不定詞の副詞的用法 など高頻度項目
- マスタに存在しない現象が検出された場合は `unclassified` として記録し、教師ダッシュボードで確認できるようにする（マスタ拡充のフィードバックループ）

- **すべてのプロンプトは `ZDP_プロンプト設計.md` に定義する。実装コードにプロンプト本文を書かないこと**

## 9. 非機能要件

- **コスト**: 1診断セッションあたりのAPI呼び出しを平均10回以下、キャッシュヒット時は3回以下に抑える
- **個人情報**: 氏名・メールアドレスを保存しない。学習者IDは出席番号ベースの匿名ID
- **応答速度**: 各プローブ提示まで3秒以内（Flash の速度を活かす）
- **可用性**: スプレッドシート書き込み失敗時もセッションは継続し、ローカルに退避して再送

## 11. 受け入れ基準（MVP）

1. 教科書レベルの英文を入力すると、語彙難易度リストと文法項目候補が内部生成される
2. ダミーマスタの範囲内の文法項目について、診断プローブが2種類以上提示される
3. プローブ回答（正誤×確信度）から4象限分類が行われ、Error候補が特定される
4. Error項目に対し、検証済みの類似例文が i+1 難易度で出題される
5. 同一英文の2回目の入力ではキャッシュが使われ、STEP 1-2 のAPI呼び出しが発生しない
6. すべてのログがスプレッドシートに記録される

### ZDPアプリの使い方

`http://localhost:3000`

#### 生徒フロー
1. **英文入力**: 分からない英文と現在レベル（A1〜B2）を入力（任意でクラスコード＋出席番号）
2. **語彙チェック**: CEFR-J照合で検出した難単語のうち「意味を知っている語」をタップ（LLM不使用）
3. **確認プローブ**: 表層パターンで絞り込んだ文法項目について、4択＋確信度(1〜5)で回答
4. **診断結果**: 「あなたの躓きは○○の可能性が高い」＋足場（前提項目）を提示
5. **類似例文で練習**: i+1難易度の検証済み例文＋問い返し型フィードバック（ヒントは段階的にエスカレート）

#### 教師フロー（`/teacher`）
- 現在のマスタの状態確認
- **T-02**: マスタアプリからエクスポートした `grammar_master.json` をアップロードして差し替え（スキーマ検証・バージョン照合付き）

#### 内部処理（ブレ最小化アーキテクチャ）
`STEP1 前処理（プログラム）` → `STEP2 P-01タグ付け（self-consistency 3回多数決）` → `STEP3 P-02プローブ` → `STEP4 4象限分類` → `STEP5 P-03生成＋P-04検証＋P-05フィードバック`。同一英文の2回目はキャッシュヒットし、STEP1-2のAPI呼び出しはゼロになります。

### マスタアプリの使い方と日常運用フロー

`http://localhost:3100`

| 画面 | 機能 |
|------|------|
| ダッシュボード | ステータス別・レベル別カバレッジ、エクスポート/インポート/CSV |
| 新規項目・項目編集 | スキーマ準拠フォーム。陽性例のspanは英文を選択して「選択→span」で指定。複製・削除 |
| クイックメモ | スマホから30秒で `status: memo` 登録（授業中用） |
| テストベンチ | 登録例文をP-01で判定し正答率を測定。誤判定の詳細（votes/evidence）を表示。verified昇格 |
| 前提グラフ | prerequisitesの依存グラフ（Mermaid）。孤立項目検出、学習経路ハイライト、クリックで編集へ |

#### 日常運用フロー
```
【授業中】生徒の躓きに気づく → クイックメモ登録（memo）
【放課後】memoを清書 → 判定基準・陽性例3・陰性例2・誤解・プローブ雛形を記述（draft）
        → テストベンチ実行
            正答率 < 90% → 誤判定を分析（B-03）→ 記述を修正 → 再テスト
            正答率 ≥ 90% かつ runs ≥ 10 → verified に昇格
        → 前提グラフで位置づけ確認・prerequisites設定 → published に昇格
【月1回】published全件をエクスポート → ZDPアプリへ供給 / 一括回帰テスト（B-06）
```

昇格条件（自動チェック）: **陽性例3件以上・陰性例2件以上・テスト正答率90%以上・runs 10回以上**。満たさない項目はverified/publishedにできません。
