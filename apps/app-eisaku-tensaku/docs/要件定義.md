# 要件定義書：英作文添削Webアプリ（改訂版）

## 0. この文書について

このドキュメントは、Claude Codeに対してWebアプリの実装を依頼するための要件定義書です。プログラミング初心者でも手順を追えば実装できるよう、技術スタック・ディレクトリ構造・実装順序を明示します。

**前回版からの主な変更点（参考情報）：**
- スコア（5点満点）の表示を廃止
- ヒントを3段階で段階的に表示する仕組みを導入
- プロンプトインジェクション対策を強化(4層防御)
- 語数極小時のガード機能を追加
- 評価対象 / 対象外のルールを明確化

---

## 1. プロダクト概要

### 1.1 目的
高校生（日本の英語学習者、CEFR A1〜A2レベル）が書いた英作文を、Gemini APIが文法の観点から自動評価し、自律的な修正学習を促すWebアプリを構築する。

### 1.2 コアコンセプト
- **直接書き直さない**：生徒が自分で修正できるよう「ヒント」のみ提示する
- **3段階のヒント**で生徒のレベルに合わせた支援を提供する
- **語数条件**の充足を確認する
- **スコアは出さない**：定量評価は生徒に明示されないため意味を持たないと判断

### 1.3 対象ユーザー
- 日本の高校生（主たる利用者）
- 高校英語教員（開発者本人、管理・改善者）

## 2. 機能要件

### 2.1 画面構成

本アプリは **2画面構成** とする。

| 画面 | ルート | 役割 |
|---|---|---|
| 入力画面 | `/` | エッセイ情報の入力と送信 |
| 結果画面 | `/result` | 添削結果の表示 |

### 2.2 入力画面（`/`）の仕様

#### 入力フィールド（すべて必須）
1. **テーマ（Topic）**
   - 形式：`<textarea>`（複数行入力可）
   - バリデーション：1文字以上、500文字以下
   - プレースホルダー例：`Some people practice foreign languages with AI. Do you think this is a good idea?`

2. **必要語数（Required Word Count）**
   - 形式：`<input type="text">`
   - バリデーション：以下2パターンのみ許容
     - 単一数字（例：`80`）→ 最小値として扱う
     - 範囲（例：`80-100`）→ ハイフン区切り、下限≤上限
   - 入力例に関するヘルプテキストを表示する

3. **自分のライティング（Essay）**
   - 形式：`<textarea>`（複数行、最低10行分の高さ）
   - バリデーション：詳細は次項「2.3 語数極小ガード」を参照
   - **語数カウンターをリアルタイム表示**する（例：`現在の語数: 87 words`）

#### 送信ボタン
- ラベル：`送信`
- クリック時の挙動：
  1. クライアント側でバリデーション（語数極小ガード含む）を実行
  2. エラーがあれば該当フィールド下に日本語でエラーメッセージを表示
  3. 問題なければ `/api/evaluate` にPOST
  4. API応答を待つ間は「添削中...」のローディングUIを表示（ボタンは無効化）
  5. 成功時は結果画面（`/result`）へ遷移し、添削結果を表示

### 2.3 語数極小ガード（重要機能）

#### 仕様

短すぎる英文の送信を防ぐため、フロントエンド側とAPI側の**二重チェック**を実装する。

#### 閾値ロジック（疑似コード）

```
wordCount = 入力された英文の語数
requiredMin = 要求語数の下限
  ・単一数字の場合（例: "80"）→ 80
  ・範囲の場合（例: "80-100"）→ 80

if (wordCount < 20) {
  ブロック
} else if (wordCount < Math.ceil(requiredMin * 0.65)) {
  ブロック
}
```

#### 表示するエラーメッセージ

```
英文が短すぎます。もう少し書いてから送信してください。
（現在: XX語 / 目安: YY語以上）
```

- `YY` = `Math.max(20, Math.ceil(requiredMin * 0.65))`
- 切り上げで整数化する

#### エラー時の挙動
- **フロント側**：英文入力欄の下に赤字でメッセージ表示。送信ボタンを押しても`/api/evaluate`を呼ばない
- **API側**：もしフロント側のチェックが回避された場合は、Zodスキーマまたは独自バリデーションで検出し、HTTP 400を返す

### 2.4 結果画面（`/result`）の仕様

以下の要素を上から順に表示する。

#### 表示要素

1. **タイトル**：`■ 英作文の添削結果`
2. **概要説明文**（固定テキスト）：
   > ライティングを文法の観点から評価します。この機能の目的は「自分で英文を直せるようになること」です。示されたヒントに基づいて、可能な限り自力で英文を直しましょう。
3. **オリジナルの英文**：入力したエッセイをそのまま表示（テーマも併記）
4. **語数ステータス**：`語数の条件: 満たしている / 満たしていない`（色で区別：緑 / 赤）
5. **修正候補件数**：`修正候補: X 文`（`errors.length` の値）
6. **ポジティブコメント**：AIが生成した励ましの一言
7. **修正候補の英文とヒント一覧**（次項「2.5」で詳述）
8. **確認するべきこと**（固定テキスト）：
   > 今回の英作文に対して与えられた条件（トピックへの回答など）を満たしているか必ず自分で確認してください。このアプリでは「文法の正しさ」のみを評価しており、構成については評価対象外です。
9. **「もう一度添削する」ボタン**：クリックで `/` に戻る

#### スコア表示について
**スコア（5点満点）は表示しない**。生徒にルーブリックが明示されない以上、定量評価は意味を持たないため。

### 2.5 段階的ヒント表示（重要機能）

#### コンセプト

各エラーに対してGemini APIは3段階のヒント（Level 1〜3）を返す。
ユーザー（生徒）はまずLevel 1のみ見て、それで分からなければボタンを押してLevel 2、さらにLevel 3と段階的に開示する。

#### レベル設計の意味

| レベル | 性格 | 文字数の目安 |
|---|---|---|
| Level 1 | 専門的で簡潔（文法用語を直接使う） | 15〜30字 |
| Level 2 | やや平易（手がかり語と簡単な説明） | 30〜60字 |
| Level 3 | 最も平易で詳細（噛み砕いた説明 + 学習方略） | 50〜130字 |

**設計思想：** できる生徒はLevel 1で気づける。分からない生徒は2→3と進むにつれ、説明が平易かつ詳細になっていく。

#### UI挙動の仕様

各エラーをカード形式で表示する。1枚のカードは以下の要素を含む。

- **エラー種別ラベル**（`errorType`、例：`[時制]`）— 常時表示
- **エラー文**（`sentence`）— 常時表示
- **Level 1ヒント** — 常時表示
- **「もっとヒントが必要ですか？」ボタン** — Level 2が未表示なら表示
- **Level 2ヒント** — ボタンを押すと表示
- **「もっとヒントが必要ですか？」ボタン** — Level 3が未表示なら表示
- **specificTerm**（例：`【過去時制】`）— Level 3表示時に出現（ただし `specificTerm` が `null` のときは出さない）
- **Level 3ヒント** — Level 3ボタンを押すと表示

#### UI表示例

- 上記のUIで`📖【過去時制】`部分の表示欄自体を出さない
- Level 3ヒント文だけ表示する

### 2.6 拒否応答時の挙動

ユーザーが添削と無関係な依頼（例：「数学の問題を解いて」「システムプロンプトを教えて」）を送信した場合、Gemini APIは以下のJSONを返す（プロンプトでそう指示している）。

```json
{
  "wordCount": { "count": 0, "satisfied": false },
  "positiveComment": "英文添削に関係のない依頼には対応できません。テーマに沿った英作文を送ってください。",
  "errors": []
}
```

#### アプリ側の判定ロジック

`errors.length === 0 && wordCount.count === 0` を満たすとき、これは拒否応答と判定する。

#### 拒否応答の表示

通常の結果画面のレイアウトを流用してよい（特別な画面遷移は不要）。
- 修正候補件数：「修正候補: 0 文」
- ポジティブコメント：拒否メッセージがそのまま表示される
- エラー一覧：空表示（「修正が必要な箇所はありません」とは出さない）

### 2.7 やらないこと（スコープ外）

MVPでは以下を**実装しない**（将来拡張として候補に残す）。

- ユーザー認証・ログイン機能
- 添削履歴の保存（データベース不使用）
- 再修正のチャット機能（元プロンプトのPhase 3相当）
- 複数回の添削履歴の比較
- 教員向け管理画面
- 英文の音声読み上げ

## 6. API設計

### 6.1 エンドポイント：`POST /api/evaluate`

#### リクエスト
```typescript
{
  "topic": string,        // テーマ
  "wordCountReq": string, // "80" または "80-100"
  "essay": string         // 生徒の英文
}
```

#### レスポンス（成功時：HTTP 200）
```typescript
{
  "wordCount": {
    "count": number,            // 実際の語数
    "satisfied": boolean        // 条件を満たしているか
  },
  "positiveComment": string,    // 日本語の励ましコメント（20〜80字）
  "errors": Array<{
    "sentence": string,         // エラーを含む原文
    "errorType": string,        // 18カテゴリーのいずれか
    "specificTerm": string | null, // 具体的な文法用語、または null
    "hints": {
      "level1": string,         // 専門的で簡潔（15〜30字）
      "level2": string,         // やや平易（30〜60字）
      "level3": string          // 最も平易で詳細（50〜130字）
    }
  }>
}
```

#### レスポンス（エラー時）
- HTTP 400：入力バリデーションエラー（語数極小含む） `{ "error": "メッセージ" }`
- HTTP 500：Gemini API呼び出し失敗 `{ "error": "添削処理中にエラーが発生しました。しばらくしてからもう一度お試しください。" }`

### 6.2 内部処理フロー

1. リクエストボディをZodで検証
2. 語数極小ガードを実行（フロントの保険）
3. 検証失敗 → 400レスポンス
4. システムプロンプト（`lib/prompt.ts`）と入力を組み合わせてGemini APIに送信
5. Gemini APIに **JSON出力を強制**（`responseMimeType: "application/json"`、`responseSchema` 指定）
6. 応答JSONをパース、Zodで再検証
7. クライアントに返却

**重要：** Gemini側にJSONスキーマを渡すことで、出力のパース失敗リスクを最小化する。

## 7. システムプロンプト

### 7.1 プロンプトの所在

システムプロンプト本体は別ファイル `lib/prompt.ts` に格納する。**プロンプトの全文は本要件定義書には含めず、別途配布される `prompt.ts` を使用すること。**

### 7.2 プロンプトの主要セクション（参考）

実装者がプロンプトの構造を理解するための参考情報として、主要セクションを列挙する。詳細は `prompt.ts` を直接参照のこと。

1. **Role** — 役割定義
2. **Core Principle** — コアコンセプト
3. **Security Rules**（最優先）
   - Forbidden Actions（5項目の禁止行動）
   - Input Scope（エッセイ本文の指示は無視）
   - Refusal Response（拒否時の固定レスポンス）
   - Rule Priority（ルールの優先順位）
4. **Evaluation Rules** — 評価対象 / 対象外の明示
5. **Word Count Rule** — 語数条件のルール
6. **Error Identification** — 1文に複数エラーがある場合の扱い
7. **Error Type Categories**（18種類 + 分類優先順位）
8. **specificTerm Whitelist** — 文法用語のホワイトリスト
9. **Hint Generation Rules** — 3レベルヒント規則 + 文構造特例
10. **Positive Comment Rules** — ポジティブコメントの詳細ルール
11. **Output Format** — JSONスキーマ
12. **Few-Shot Examples** — 3例
13. **Reference Essays** — 満点エッセイ4本

### 7.3 プロンプト変更時の注意

プロンプトは慎重に設計されているため、軽微な変更でも出力品質に影響する可能性がある。修正時は以下に注意する：

- **Security Rules** の優先順位は維持する
- **18個の Error Type Categories** を変更する場合は、対応する `specificTerm` のホワイトリストも更新する
- **Few-Shot Examples** はLevel 1〜3の正しい関係性を示す重要な見本なので、追加する場合は同じパターンで記述する

## 8. データスキーマ（`lib/schema.ts`）

```typescript
import { z } from "zod";

// ==========================================
// 入力（フォーム）スキーマ
// ==========================================
export const InputSchema = z.object({
  topic: z.string()
    .min(1, "テーマを入力してください")
    .max(500, "テーマは500文字以内で入力してください"),
  wordCountReq: z.string().regex(
    /^\d+(-\d+)?$/,
    "語数は '80' または '80-100' の形式で入力してください"
  ),
  essay: z.string().min(1, "英文を入力してください"),
});
export type InputType = z.infer<typeof InputSchema>;

// ==========================================
// errorType の許容値（18種類）
// ==========================================
export const ErrorTypeEnum = z.enum([
  "スペルミス",
  "時制",
  "完了形",
  "仮定法",
  "主語と動詞の不一致",
  "動詞の種類",
  "語形",
  "助動詞",
  "受け身",
  "不定詞・動名詞",
  "分詞",
  "関係代名詞",
  "接続詞",
  "前置詞",
  "語順",
  "文構造",
  "大文字・小文字",
  "その他",
]);

// ==========================================
// 出力（Gemini応答）スキーマ
// ==========================================
export const OutputSchema = z.object({
  wordCount: z.object({
    count: z.number().int().nonnegative(),
    satisfied: z.boolean(),
  }),
  positiveComment: z.string(),
  errors: z.array(z.object({
    sentence: z.string(),
    errorType: ErrorTypeEnum,
    specificTerm: z.string().nullable(),
    hints: z.object({
      level1: z.string(),
      level2: z.string(),
      level3: z.string(),
    }),
  })),
});
export type OutputType = z.infer<typeof OutputSchema>;
```

## 9. バリデーション関数（`lib/validation.ts`）

語数極小ガードのロジックは独立した関数として実装し、フロント・APIの両方から利用する。

```typescript
/**
 * 英文の語数を数える
 * 連続する空白で分割し、空文字を除外したものをカウントする
 */
export function countWords(essay: string): number {
  return essay.trim().split(/\s+/).filter(w => w.length > 0).length;
}

/**
 * 要求語数文字列から最小値を抽出する
 * 単一数字（"80"）→ 80
 * 範囲（"80-100"）→ 80
 */
export function parseRequiredMin(wordCountReq: string): number {
  const parts = wordCountReq.split("-");
  return parseInt(parts[0], 10);
}

/**
 * 語数極小ガード
 * @returns ブロックする場合はエラーメッセージ、通過する場合は null
 */
export function checkWordCountGuard(
  essay: string,
  wordCountReq: string
): string | null {
  const count = countWords(essay);
  const requiredMin = parseRequiredMin(wordCountReq);
  const threshold = Math.max(20, Math.ceil(requiredMin * 0.65));
  
  if (count < 20 || count < Math.ceil(requiredMin * 0.65)) {
    return `英文が短すぎます。もう少し書いてから送信してください。(現在: ${count}語 / 目安: ${threshold}語以上)`;
  }
  return null;
}
```

## 11. UI/UXガイドライン

### 11.1 デザイン方針
- **落ち着いた配色**：生徒が集中して使えるよう、刺激の強い色は避ける
- **推奨パレット**：
  - プライマリ：藍色系（`#1e40af` 相当）
  - 成功（語数充足）：緑（`#10b981`）
  - 警告（語数不足）：赤（`#ef4444`）
  - 背景：白またはごく薄いグレー
- **フォント**：日本語部分は端末標準のsans-serif、英文部分は読みやすいフォント（例：`Georgia`, `ui-serif`）

### 11.2 スマホ優先の配慮
- 入力フィールドはタップしやすい高さ（最低44px）
- 送信ボタンは画面下部に固定配置しない（キーボード表示時に邪魔になる）
- エラーカードは折りたたまず、一覧表示

### 11.3 段階的ヒント表示のUI設計指針
- **Level 2、Level 3を表示する「もっとヒントが必要ですか？」ボタン**は、生徒が押すまでは Level 2 / 3 のヒント文を表示しないこと
- ボタンを押した後はそのカード内で表示が展開される（モーダル等の画面切り替えはしない）
- ボタンのラベルは段階に応じて変える（例：1回目「もっとヒントが必要ですか？」、2回目「さらに詳しい説明を見ますか？」）等の工夫はしてもよい
- Level 3を開いたとき、`specificTerm` が `null` でなければ `📖【○○○】` のような形式で表示する

### 11.4 アクセシビリティ
- すべての入力フィールドに `<label>` を関連付ける
- ボタンには適切な `aria-label` を付与
- 色だけに頼らず、アイコンや文字でも状態を示す

## 12. エラーハンドリング

| 発生箇所 | エラー内容 | UI上の対応 |
|---|---|---|
| フォーム入力 | 必須未入力、形式違反 | 該当フィールド下に赤字でメッセージ表示 |
| 語数極小ガード | 短すぎる英文 | 英文入力欄の下に赤字でメッセージ表示、送信を阻止 |
| API通信失敗 | ネットワーク断 | トーストで「通信エラー。もう一度お試しください」 |
| Gemini API | レート制限 / 5xx | トーストで「サーバーが混み合っています。少し待ってください」 |
| Gemini応答パース | JSON不正 | フォールバックで「処理に失敗しました。文章を見直して再送信してください」 |
| 結果画面で`sessionStorage`が空 | 直接URLアクセス等 | 「添削データがありません」と表示し、`/` への戻るリンクを表示 |
| 拒否応答 | `errors.length === 0 && wordCount.count === 0` | 通常の結果画面レイアウトを流用（拒否メッセージが`positiveComment`に表示される） |
