# 修正案 — 総合発表会サイト 課題洗い出しと改善提案

作成日: 2026-07-04
対象: リポジトリ全体（静的サイト / Apps Script / ドキュメント / teacher-guide.html）

---

## 0. 現状構成の整理（理解サマリ）

```
生徒: Google Slides を Classroom に提出
  ↓ ① classroom.gs  … 提出物のファイルIDを「生徒対応」シート経由でグループに紐付け
  ↓ ② drive.gs      … Slides → PDF 変換、Drive に保存、「組織内のみ」共有、URLをシートに記録
  ↓ ③ sheets.gs     … シート → schedule.json 生成
  ↓    github.gs    … GitHub Contents API で data/schedule.json を直接コミット
  ↓ Vercel が自動デプロイ
閲覧者: index.html（一覧・検索・ソート）→ group.html（詳細・PDF埋め込み）
```

Google Workspace 中心の運用に置き換えたことで属人性が大きく下がっており、方向性は良いです。以下は「残っている穴」と「さらにシンプル・安全にできる点」です。

## A. セキュリティ上の課題（優先度: 高）

### A-1. GitHub トークン（classic / repo 権限）をスプレッドシートに平文保存 ★最重要

- **現状**: `apps-script/config.gs` の設計上、「設定」シートの B 列に PAT を貼る。ガイドtoo `repo` スコープの classic トークンを指示している。
- **問題**:
  - `repo` スコープの classic PAT は **そのアカウントの全リポジトリへの書き込み権限**を持つ。シートの閲覧権限がある教員・関係者全員がトークンを読める。
  - シートの共有範囲を広げた瞬間（引き継ぎ時にやりがち）にトークンが漏れる。
- **修正案**:
  1. **Fine-grained PAT に変更**: 対象リポジトリ1つだけ・権限は `Contents: Read and write` のみ・有効期限を設定（例: 1年、年度更新時に再発行）。
  2. **保存先を Script Properties に変更**: シートのセルではなく `PropertiesService.getScriptProperties()` に保存する。Apps Script エディタの「プロジェクトの設定 → スクリプト プロパティ」から GUI で入力でき、シート閲覧者には見えない。
     - `sheets.gs` の `getSettings()` を「シート → Script Properties の順で探す」形にすれば互換を保てる。
     - メニューに「🔑 GitHub トークンを設定」を追加し、プロンプト入力 → Script Properties 保存にすると教員でも迷わない。
  3. ガイド類（`apps-script/README.md` / `teacher-guide.html`）のトークン取得手順を fine-grained PAT の手順に差し替える。

### A-2. 実際の生徒 PDF が公開リポジトリ / Vercel 上に存在

- **現状**: `sample/report-101.pdf`・`sample/slides-101.pdf` がリポジトリにコミット済みで、Vercel から**認証なしで誰でも閲覧可能**（コミット `7e3d7bb`「実PDFを埋め込み」）。未追跡の `sample data/` にも Drive からダウンロードした実データ（スライド.pdf / レポート.pdf）がある。
- **問題**: 「PDF は Drive に置き、組織内のみ共有」という設計原則を自ら破っている。生徒名等が含まれていれば個人情報の公開に当たる。
- **修正案**:
  1. 実データ由来の PDF なら `git rm` + **git 履歴からも削除**（`git filter-repo` 等）し、`data/schedule.json` の 101 班のリンクを Drive のダミー or 空に戻す。
  2. デモ用に埋め込みを見せたいなら、**架空の内容で自作したダミー PDF** に差し替える。
  3. `sample data/` はコミットしない。`.gitignore` に `sample data/` を追加。
  4. 完了後、`vercel.json` の `/sample/(.*\.pdf)` ヘッダー設定は不要になるので削除可（ダミーPDFを残すなら維持）。

### A-3. schedule.json の URL をそのまま iframe / リンクに使用

- **現状**: `group.js` が `pdf_embed_url` 等を `escapeHtml` のみで `iframe src` / `a href` に流し込む。HTML エスケープでは `javascript:` 等の危険スキームは防げない。
- **問題**: データ源はシート（教員入力）なので直ちに危険ではないが、シート編集権限者が増える運用なので防御層を足すべき。
- **修正案**: `group.js` に URL 検証関数を追加し、許可するものだけ描画する。

  ```js
  function isSafeUrl(u) {
    if (!u) return false;
    if (u.startsWith('/')) return true; // 同一オリジン
    try {
      var parsed = new URL(u);
      return parsed.protocol === 'https:' &&
        (parsed.hostname === 'drive.google.com' || parsed.hostname === 'docs.google.com');
    } catch (_) { return false; }
  }
  ```

### A-4. セキュリティヘッダーの不足

- **現状**: `vercel.json` は `X-Content-Type-Options` と `X-Frame-Options` のみ。
- **修正案**: 以下を追加。

  ```json
  {"key": "Content-Security-Policy",
   "value": "default-src 'self'; frame-src https://drive.google.com https://docs.google.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; img-src 'self' data:; script-src 'self'"},
  {"key": "Referrer-Policy", "value": "no-referrer"},
  {"key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()"}
  ```

  ※ teacher-guide.html はインライン `<style>` のみなので `style-src 'unsafe-inline'` が必要。CSP 導入時はローカルで全ページの表示確認をすること。

### A-5. teacher-guide.html が本番サイトに公開される

- **現状**: ルート直下にあるため、Vercel にデプロイすると `https://…/teacher-guide.html` で誰でも閲覧できる。秘密情報は含まないが、管理フロー・GitHub 利用・シート構成など内部運用情報が全世界に見える。
- **修正案**（いずれか）:
  - **案1（推奨）**: `.vercelignore` を作成して `teacher-guide.html` と `docs/` `apps-script/` `scripts/` `sample/` を除外する（リポジトリには残るのでデプロイのみ除外）。
  - **案2**: ガイド HTML を Google ドライブ（教員用共有ドライブ）に置き、リポジトリでは docs/ に保管。「誰でも扱える」目的なら、教員が最初に見る場所は Drive の方が自然。

## B. バグ・不整合（優先度: 高〜中）

### B-1. `eventDate` / `notice` が JSON から反映されない

- `index.html` の `#eventDate` は「2025年3月13日」と**ハードコード**。`schedule.js` は `data.eventDate` / `data.notice` / `eventTitle` を一切使っていない（旧 `viewer/schedule.js` にはあった処理が新デザイン移行で消失）。
- **影響**: シートの「Event Date」を更新してもサイトの日付が変わらない。毎年の運用で必ず踏むバグ。
- **修正案**: `schedule.js` のロード完了時に `#eventDate` へ `data.eventDate` を、ナビ/ヒーローのタイトルへ `data.eventTitle` を設定。`notice`（サンプルデータ注意書き等）の表示欄も設ける。

### B-2. ヒーローの統計「3 時間帯」がハードコード

- `index.html:34` で `3` 固定。時間帯数はデータから導出できる（`fillFilterOptions` で既に集計済み）。グループ数・教室数と同様に JS で埋める。

### B-3. `group.html` が `schedule.js` を不要に読み込んでいる

- `group.html:35-36` で `schedule.js` + `group.js` の両方をロード。`schedule.js` はガード条件で何もしないが、無駄で紛らわしい。
- **修正案**: `group.html` から `schedule.js` を外す。`group.js` の `fetch('data/schedule.json')` は相対パスなので、`schedule.js` と同じ `getBasePath()` 方式に揃える（サブパス配信でも壊れない）。共通処理（`escapeHtml` / `loadSchedule` / `getBasePath`）は `common.js` に切り出すとさらに整理できる。

### B-4. 「⚡ すべて一括実行」が実際には全自動ではない

- `main.gs` の `runFetchSubmissions` 等は各ステップ完了時に `ui.alert`（「次に②を実行してください」）を出す。`runAll` から呼ばれた場合も**毎回 OK クリックが必要**で、メッセージ内容も個別実行前提で不整合。ガイドの「押すだけで3〜5分待つ」という記述と食い違う。
- **修正案**: 各 `run*` 関数に `silent` 引数（または内部フラグ）を追加し、`runAll` 実行中は途中のアラートを抑止して最後に1回だけ結果を表示する。`runAll` の空 `catch` にも `Logger.log(e)` を入れる。

### B-5. 生徒の紐付けが「表示名」キーで同姓同名・表記ゆれに弱い

- `getStudentMapping()` は Classroom の**フルネーム文字列**をキーにする。同姓同名で衝突し、姓名間スペースの全角/半角違いでも不一致になる。
- **修正案**: Classroom の `userProfiles` から **メールアドレス**を取得し、「生徒対応」シートを「メールアドレス ↔ group_id」に変更する（学校配布アカウントなら一意）。移行が大変なら、少なくとも照合時に空白を正規化し、不一致者一覧をシートに書き出す「照合レポート」列を追加する。

### B-6. 再提出が反映されない（PDF 変換のスキップ条件）

- `drive.gs` は URL セルが埋まっている行を**スキップ**するため、生徒が再提出しても古い PDF のまま。
- **修正案**: メニューに「♻️ 指定グループのPDFを再変換（URL列をクリアして再実行）」を追加、またはファイルの `getLastUpdated()` と比較して新しければ再変換。最低限、teacher-guide の FAQ に「再提出時は該当行の URL 列（I〜L列）を消して②を再実行」と明記する。

### B-7. `docs/DATA_SCHEMA.md` の「列の並び順は自由」が現行実装と矛盾

- Apps Script 移行後は `config.gs` の `COL` が**固定列番号**で参照するため、列の並び替え・挿入は壊れる。DATA_SCHEMA.md は旧 CSV エクスポート（ヘッダー名参照）前提の記述のまま。
- **修正案**: DATA_SCHEMA.md を「Apps Script 運用ではグループ一覧シートの列順は固定（A〜L）。変更する場合は config.gs の COL を合わせる」に書き換える。（あるいは `getGroupList` をヘッダー名参照に変えて本当に順序自由にする。堅牢性ではこちらが上）

### B-8. 表記ゆれ「総合探", "探求 / 探究」

- サイト・ガイド・テンプレートは「総合**探求**成果報告会」、README・sample は「総合**探究**」。学習指導要領上は「総合的な**探究**の時間」。
- **修正案**: 正式名称を1つ決めて全ファイル一括置換（`index.html` / `group.html` / `group.js` / `teacher-guide.html` / `template.gs` の `DESIGN.footerText` / `main.gs` の初期値 など）。

### B-9. group_id の例がドキュメントごとにバラバラ

- 実データ: `101`、teacher-guide: `1-01`、README: `201-1`。教員が最初に迷うポイント。
- **修正案**: 実データに合わせて全ドキュメントの例を統一（例: `101` = 1年1組1班 方式なら、その命名規則も1行書く）。

## C. 構造の簡素化（優先度: 中）

### C-1. 旧世代の残骸を削除

| 対象 | 理由 |
|------|------|
| `viewer/`（index.html, schedule.js, style.css） | ルートのサイトと重複した旧バージョン。混乱の元 |
| `presenter/`（空ディレクトリ） | presenter 機能は削除済み（コミット `edd07a1`）。README からも記述を削除 |
| `docs/PRESENTER_GOOGLE_SLIDES.md` | presenter 前提のドキュメント。機能が無いのに残っている |
| `group.js` 内の `slides_view_url` / `slides_present_url` 参照 | presenter 時代の名残（`group.js:71,84-86,91`）。schedule.json 生成側（sheets.gs）はこれらを出力しない |
| `scripts/fix-schedule-keys.js` | 「1回限り実行用」と自記している移行スクリプト。役目終了 |
| `scripts/sheets-to-json.js` | Apps Script の③で代替済み。残すなら「Apps Script が使えない時のバックアップ手段」と README に一言書く |
| `.DS_Store`（git 追跡中） | `.gitignore` にはあるが追跡済み。`git rm --cached .DS_Store` |
| `docs/teacher-guide.html`（削除済み・未コミット） | ルート移動をコミットして状態を確定させる |

### C-2. README.md の全面書き直し

現 README は presenter のパスワード運用・手動 JSON 貼り替え・CSV エクスポートなど**旧運用と新運用が混在**しており、「誰でも扱える」ためのドキュメントとして機能していません。

- **構成案**:
  1. これは何か（3行）
  2. 教員向け → **teacher-guide.html を見る**（1行で誘導）
  3. リポジトリ構成（現状に合わせたツリー）
  4. 開発者向け: ローカル確認方法・デプロイの仕組み（詳細は docs/ へ）
- 重複している `docs/DEPLOY.md` との内容整理（デプロイ詳細は DEPLOY.md に一本化し README からリンク）。

### C-3. ホスティングの簡素化案（任意・検討）

現在は「GitHub（個人アカウント）+ Vercel（個人アカウント）」の2サービス依存で、ここが**最後に残った属人性**です。

- **案1（推奨・変更小）**: GitHub を **school 用 Organization**（または共有アカウント）に移管し、Vercel も Team もしくは共有アカウントにする。teacher-guide に「引き継ぎ対象アカウント一覧」を明記。
- **案2（依存を1つ減らす）**: Vercel をやめて **GitHub Pages** にする。静的サイトなので機能は同等、Apps Script の push → Pages 自動デプロイの流れも同じ。管理アカウントが GitHub 1つになり、引き継ぎがさらに単純になる。`vercel.json` のヘッダーは Pages では効かない点だけ注意（`<meta>` で代替、または現状維持を選ぶ）。

### C-4. フロントエンドの小整理

- `schedule.js` の検索イベントで `input` と `keyup` の二重登録 → `input` のみに。
- `sortGroups` の第2ソートキー（`timeslot_label`）だけ `dir` が掛かっていない（`schedule.js:73-74`）。動作は許容範囲だが意図を統一する。
- `escapeHtml` 等の重複定義（schedule.js / group.js）→ `common.js` へ（B-3 と同時に）。

### D-2. ガイド自体の改善

1. **モバイル時のナビ**: `@media (max-width:700px)` でサイドバーを消しているが代替ナビが無い。冒頭に目次リスト（アンカーリンク）を出す、または details 化したハンバーガー的メニューを追加。
2. **印刷対応**: 教員は紙で読む可能性が高い。`@media print`（サイドバー非表示・背景色除去・URL 展開）を追加。
3. **公開場所**: A-5 のとおり Vercel 公開から除外するか、教員用共有ドライブに置く。ガイドの位置づけ（「このファイルは校内限定資料」）を冒頭 badge 付近に一文追加。
4. HTML ガイドという選択自体は目的（MD に馴染みのない教員向け）に合っており妥当。**マスタは HTML のままでよい**が、更新担当者向けに「このファイルはどう更新するか」を末尾に1セクション追加すると引き継ぎが完結する。
