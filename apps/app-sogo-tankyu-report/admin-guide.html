はじめに

システム構成

このサイトは、次の3つのサービスを連携させた静的サイト＋自動更新パイプラインです。日常運用ではスプレッドシートだけを触り、GitHub と Vercel は「自動で動く裏方」です。

1
Google Workspace スプレッドシート ＋ Apps Script（データのマスタ）

Classroom から提出物を取得し、Drive で PDF 変換・共有設定。シートの内容から data/schedule.json を生成します。コードはリポジトリの apps-script/ にあり、スプレッドシートの Apps Script エディタに貼り付けて使います。

2
GitHub リポジトリ（サイトのソースコード置き場）

Apps Script が GitHub API 経由で data/schedule.json を直接コミットします（認証は Fine-grained PAT）。HTML/CSS/JS の修正もこのリポジトリに push します。

3
Vercel ホスティング（自動デプロイ）

GitHub への push を検知して自動でデプロイします。vercel.json でセキュリティヘッダー（CSP等）を、.vercelignore で内部資料（このガイド・docs・apps-script）の非公開を設定しています。

アクセス制御の設計（2段構え）

層しくみ性質
サイトの閲覧ゲート Google Identity Services によるログイン。ID トークンの hd/メールドメインを「設定」シートの Allowed Domain と照合（実装: common.js） クライアントサイドのゲート。一般閲覧者は確実に弾けるが、schedule.json 自体は技術的には取得可能
ファイル本体の保護 PDF は Drive に置き、共有「組織内のみ」（Apps Script が自動設定） Google の認証・認可で保護される本命の防御。リンクが漏れても組織外は開けない

個人情報や成績等のセンシティブな情報は schedule.json（グループ名・テーマ・概要）に載せない運用としてください。

初回セットアップ（1年目のみ）

① Google スプレッドシートを準備する

⚠️
初回のみの作業です 一度設定すれば、翌年以降は実務編の「毎年の運用」だけで回ります。スプレッドシートは共有ドライブに置くと、担当者が変わっても引き継げます。
  1. Google スプレッドシートを新規作成する Google Drive（推奨: 共有ドライブ）で「新規 → Google スプレッドシート」を作成。名前は「総合探究 発表会管理」など分かりやすい名前にする。
  2. Apps Script エディタを開く スプレッドシートのメニューから拡張機能 → Apps Script を選択。
  3. コードファイルを貼り付ける リポジトリの apps-script/ フォルダ内の .gs ファイルの内容をコピーして、Apps Script エディタに貼り付ける（各ファイルごとに別のスクリプトファイルを作成。ファイル名は .gs を除いた名前にする）。
  4. 「初期セットアップ」を実行する スプレッドシートに戻るとメニューに「📋 発表会サイト管理」が追加されます。
    📋 発表会サイト管理 → 🔧 初期セットアップ（シート生成） を実行。

実行すると、「設定」「グループ一覧」「生徒対応」の3シートが自動生成されます。「設定」シートには以下を入力します。

③ GitHub トークンを取得・登録する

Apps Script が data/schedule.json を GitHub にコミットするために、アクセストークンが必要です。影響範囲を最小にできる Fine-grained トークンを使います。

  1. GitHub にログインし、Settings を開く 右上のアイコンをクリック → Settings
  2. Developer settings → Personal access tokens → Fine-grained tokens 左メニューの一番下にあります。
  3. 「Generate new token」をクリックし、次のとおり設定
    Token name例: sogo-happyo-site
    Expiration1年（切れたら再発行して再登録）
    Repository accessOnly select repositories → 発表会サイトのリポジトリ1つだけを選択
    PermissionsRepository permissions → Contents: Read and write のみ
  4. 表示されたトークンをコピー ⚠ 一度しか表示されません — 必ずコピーしてから閉じる。
  5. スプレッドシートのメニューから登録する 📋 発表会サイト管理 → 🔑 GitHub トークンを設定 を開き、トークンを貼り付けて OK。

学内限定

学校の Google アカウントでログイン

このサイトは @school.ed.jp のアカウントを持つ方のみ閲覧できます。

G Google でログイン

手順1: Google Cloud で「OAuth クライアントID」を作る

  1. Google Cloud Console を学校アカウントで開く 初めての場合は画面上部で「プロジェクトを作成」（名前は「発表会サイト」など）。
  2. 「APIとサービス」→「OAuth 同意画面」を設定 User Type は「内部」を選択（学校の Workspace 組織内のみが対象になります）。アプリ名・連絡先を入力して保存。
  3. 「認証情報」→「認証情報を作成」→「OAuth クライアントID」 アプリケーションの種類は「ウェブ アプリケーション」。
  4. 「承認済みの JavaScript 生成元」にサイトのURLを追加 例: https://sogo-happyo.vercel.app（本番URL。ローカル確認もするなら http://localhost:8080 も追加）。
  5. 作成すると「クライアントID」が表示されるのでコピー 1234…….apps.googleusercontent.com のような文字列です。

手順2: 「設定」シートに2つの値を入力する

キー入力する値
Allowed Domain学校のメールドメイン（例: school.ed.jp）
Google Client ID手順1でコピーしたクライアントID

入力後に「③ サイトに反映」を実行すると有効になります。両方を空欄にして反映すれば、制限なしに戻せます。

ℹ️
クライアントIDは「秘密」ではありません クライアントIDは公開されても問題ない識別子です（トークンとは違います）。シートに入力してOKです。
⚠️
閲覧ゲートの限界を理解しておく 静的サイトのため、この制限はブラウザ側で動く「ゲート」です。一般の閲覧者は確実に弾けますが、技術のある人がスケジュールデータ（グループ名・テーマ等）を直接取得することは防げません。生徒の作品（PDF）は Drive の組織内共有が守る、という役割分担です。

引き継ぎチェックリスト

管理者が変わるときは、以下の4点を引き継げば運用を続けられます。

引き継ぐものポイント
① Google スプレッドシート 共有ドライブに置いてあれば作業不要。個人のマイドライブにある場合は、共有ドライブへ移動するかオーナー権限を後任に譲渡する。Apps Script のコードとスクリプト プロパティ（トークン）もシートに付いて引き継がれます。

④ ガイド2冊 実務編 teacher-guide.html（学年の担当教員へ）と、この管理者編 admin-guide.html（後任の技術担当者へ）。

技術FAQ（システム側のトラブル）

実務担当者から「見慣れないエラーが出た」と連絡があったときは、まずここを確認してください。

「GitHub API エラー」または「GitHub トークンが未設定です」と表示される
トークンが未登録か、有効期限切れの可能性が高いです。GitHubで新しい Fine-grained トークンを発行し（→ ③の手順）、メニューの「🔑 GitHub トークンを設定」で登録し直してください。401/403 が返る場合はトークンの Repository access / Permissions（Contents: Read and write）も確認を。
「③ サイトに反映」は成功するが、サイトが更新されない
GitHub のリポジトリで data/schedule.json に新しいコミットが入っているか確認してください。コミットが入っているのに反映されない場合は、Vercel のダッシュボード → 対象プロジェクト → Deployments でビルドエラーの有無を確認します。ブラウザキャッシュの可能性もあるので Ctrl+Shift+R も試してください。
ログイン画面が出ない／誰でも見えてしまう
「設定」シートの Allowed Domain と Google Client ID の両方が入力されているか確認し、「③ サイトに反映」を実行してください。どちらかが空だと閲覧ゲートは無効になります（data/schedule.json の auth フィールドで現在の設定値を確認できます）。
ログイン画面は出るが、正しいアカウントでも「閲覧できません」と言われる／ボタンが動かない
  • Allowed Domain の値と、実際のメールアドレスのドメイン（@より後ろ）が一致しているか確認。
  • Google Cloud の OAuth クライアントの「承認済みの JavaScript 生成元」にサイトの本番URLが登録されているか確認（未登録だとログインボタンが機能しません）。
  • ブラウザの開発者ツール（F12）の Console にエラーが出ていないか確認。
Apps Script の実行ログはどこで見られる？
スプレッドシートの 拡張機能 → Apps Script でエディタを開き、左メニューの「実行数」を開くと、各実行のログ（「生徒対応なし: ○○」等）を確認できます。PDF変換に失敗したグループの原因調査に使えます。
HTML/CSS/JS を修正したい。どうやって反映する？
リポジトリをローカルに clone し、修正して main ブランチへ push すれば Vercel が自動デプロイします。ローカル確認は npx serve -l 8080（ルートの serve.json により本番と同じURL挙動になります）。データ（data/schedule.json）は Apps Script が上書きするため、手で編集しないでください。
このガイドや docs/ が本番サイトから見えないのはなぜ？
.vercelignore で teacher-guide.html・admin-guide.html・docs/・apps-script/・*.md をデプロイ対象から除外しているためです。リポジトリには含まれるので、ガイドはファイルを直接ブラウザで開くか、リポジトリ経由で共有してください。

ガイドの更新方法

ガイドは2つの HTML ファイルに分かれています。どちらもリポジトリ直下にあり、テキストエディタで編集して push すれば更新完了です。

ファイル対象読者書いてよい内容
teacher-guide.html
（実務編）		 学年の担当教員
（誰でも）		 スプレッドシート・Classroom・サイトの見方だけ。GitHub / Vercel / トークン / JSON などの技術用語は出さない（不安にさせないため）。システム側の問題は「技術担当者に連絡」で誘導する。
admin-guide.html
（管理者編・このファイル）		 技術担当者 セットアップ・引き継ぎ・技術トラブル対応。技術用語OK。
  • どちらのファイルも .vercelignore により本番サイトには公開されません。
  • 手順（メニュー名・シート構成など）を変えた場合は、apps-script/README.md と docs/ 内の該当ドキュメントも合わせて更新してください。
  • 2つのガイドは相互リンクしています。ファイル名を変える場合は両方のリンクを直してください。
