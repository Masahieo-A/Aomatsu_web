# 信頼性向上.md — SPEAKLY を「なんちゃってAIラッパー」から実用レベルの発音評価アプリへ

> 本ドキュメントは Claude Code がバイブコーディングで段階的に実装できるよう、
> **現状診断 → ELSA の実態リサーチ → API 選定 → アーキテクチャ → 実装フェーズ（受け入れ基準付き）→ 検証プロトコル** の順で構成している。
> 各フェーズは独立して着手可能。上から順に実装すること。

---

## 0. 結論（TL;DR）

| 項目 | 現状 | あるべき姿 |
|---|---|---|
| 発音スコア | `Math.random()` による**完全なダミー** | Azure AI Speech **Pronunciation Assessment** による音素レベル実測スコア |
| 単語別フィードバック | ランダムに good/ok/bad を割当 | API の単語別 AccuracyScore + エラータイプ（省略/挿入/誤読） |
| 音素別精度（進捗画面） | 初回表示時にランダム生成 | 実セッションの音素スコアを蓄積・集計 |
| 会話AI | キーワードマッチの固定スクリプト | Claude API による本物の対話 + 発話の非スクリプト評価 |
| 会話スコア | ターン数 + 乱数 | 発音・流暢さ・語彙・文法の実測 |
| STT | Web Speech API（会話のみ、精度不明・非ネイティブに弱い） | Azure Speech SDK（採点と同一パイプライン） |
| TTS | ブラウザ speechSynthesis | Azure Neural TTS（フォールバックに speechSynthesis 維持） |
| APIキー管理 | なし（クライアントのみ） | Vercel Serverless Functions でトークン発行・キー秘匿 |

**最重要原則：測定できないものを測定したふりをして表示しない。** 評価できない場合は「評価不能」と表示する。これが「信頼性」の定義である。

### 1.2 使える資産（残すもの）

- UI/UX 一式（スコアリング表示、単語ピル、音素グリッド、進捗画面）— API のレスポンス構造とほぼ1:1で対応可能
- `PHRASES` のフレーズデータ（リファレンステキストとしてそのまま採点に使える）
- localStorage の状態管理・ストリーク・XP 周り

### 1.3 インフラ上の注意点

- `vercel.json` の CSP が `connect-src 'self'` のため、**外部 API 呼び出しは現状ブロックされる**。Azure 導入時に `connect-src` へ `https://*.cognitiveservices.azure.com wss://*.stt.speech.microsoft.com https://*.tts.speech.microsoft.com` 等の追加が必須（Phase 1 参照）
- 静的サイトだが Vercel なので `api/` ディレクトリを置くだけで Serverless Functions が使える（フレームワーク不要）

## 2. リサーチ: 実際の ELSA は何をしているか

- **自社開発の ASR**: ELSA は Google/Amazon の汎用 STT を使っていない。**非ネイティブ（訛りあり）音声に特化した独自の深層学習 ASR** を、195か国の学習者音声という大規模な訛りコーパスで学習させている。汎用 STT は「何を言ったか」を当てるために訛りを吸収・正規化してしまうが、発音評価では訛り（ズレ）こそが検出対象であり、目的が正反対。ここが「STT の結果と正解文字列を比較するだけのラッパー」と本物の決定的な差
- **音素レベルの照合**: 発話を音素単位でネイティブ話者モデルと比較し、どの音素がどうズレたか（置換・脱落・挿入）を特定して矯正提案を返す
- **多面的スコア**: 発音だけでなく流暢さ・強勢（word stress）・イントネーションを個別に採点。スクリプト読み上げ（scripted）と自由発話（unscripted）の両方に対応
- **ELSA API**: B2B 提供あり（要商談・料金非公開）。個人開発の第一選択にはならない

**含意**: 個人開発で ELSA の ASR は再現できないが、**同じ方式（音素レベル強制アラインメント + Goodness of Pronunciation 系スコア）を商用 API として提供しているものを使えば、評価の信頼性は担保できる**。それが次章。

## 3. API 精査・選定

### 3.1 発音評価 API 比較

| API | 音素レベル | プロソディ | 非スクリプト評価 | 料金 | ブラウザSDK | 判定 |
|---|---|---|---|---|---|---|
| **Azure AI Speech Pronunciation Assessment** | ○（IPA出力可、音節も） | ○（en-USのみ: イントネーション・リズム・強勢） | ○（トピック関連性・語彙・文法まで） | STT 約$1/h + 評価アドオン $0.30/h。**Free tier (F0) 月5時間無料** | ○ 公式 JS SDK（`microsoft-cognitiveservices-speech-sdk`） | **採用** |
| Speechace | ○（特許技術、人間評価者と相関 0.9 を公表） | △ | ○（IELTS/PTE 推定） | サブスク制、無料枠小 | REST のみ | 次点。試験対策系に強い |
| SpeechSuper | ○（8言語） | ○ | ○ | 要問合せ | REST | 中国系。日本人学習者向け実績あり |
| Language Confidence | ○（US/UK nativeness） | △ | ○（文法・語彙・関連性） | 要問合せ | REST | 候補 |
| ELSA API | ◎ | ◎ | ◎ | **B2B・要商談** | ✕ | 個人開発では非現実的 |
| OSS 自前（wav2vec2 GOP / Kaldi） | ○ | ✕ | ✕ | GPU サーバ代 | ✕ | 検証コスト大。今回は不採用 |

### 3.2 Azure を採用する理由

1. **人間の採点との整合を Microsoft が検証・文書化している**（Transparency Note / Characteristics and limitations が公開されており、評価の限界を仕様として引用できる = 信頼性の根拠を明示できる）
2. 返却 JSON が現 UI とほぼ 1:1 対応:
   - `PronScore`（総合）→ スコアリング
   - `AccuracyScore` → 発音バー
   - `FluencyScore` → 流暢さバー
   - `ProsodyScore` → イントネーションバー
   - `CompletenessScore` → 完全性バー
   - `Words[].AccuracyScore` + `ErrorType`（Omission/Insertion/Mispronunciation）→ 単語ピル
   - `Words[].Phonemes[].AccuracyScore`（IPA 指定可）→ 音素グリッド
3. ブラウザ直結の公式 JS SDK があり、静的 HTML 構成を維持したまま導入できる（トークン発行のみサーバ側）
4. 無料枠（F0: 月5時間）で開発・検証が完結する
5. 同一キーで Neural TTS・通常 STT も使え、会話機能のスタックを統一できる

### 3.3 会話 AI

- **Claude API**（`claude-sonnet-5` 推奨、コスト重視なら `claude-haiku-4-5-20251001`）でスクリプト会話を置換
- サーバ側（Vercel Function）から呼ぶ。system prompt でトピック・ロールプレイ設定・CEFR レベル調整・「相手の英語の誤りを1つだけ優しく指摘」等を制御
- 会話中の発話は Azure の **unscripted（リファレンステキストなし）評価** に投げ、発音・流暢さ・語彙・文法スコアを実測する

## 4. 目標アーキテクチャ

```
┌────────────────────────── ブラウザ (index.html) ──────────────────────────┐
│  MediaRecorder / Azure Speech SDK (JS)                                    │
│   ├─ 発音ドリル: SDK + PronunciationAssessmentConfig(referenceText)       │
│   ├─ 会話: SDK STT → テキスト → /api/chat → 返答 → Azure TTS 再生         │
│   └─ 会話音声を unscripted 評価に並行送信                                  │
└──────────────┬────────────────────────────────────────────────────────────┘
               │ (認証トークンのみ要求。APIキーはブラウザに出さない)
```

## 5. 実装フェーズ（Claude Code 向け作業指示）

> 各フェーズの最後に「受け入れ基準」がある。**基準を満たすまで次フェーズへ進まないこと。**

### Phase 0: うそ実装の撤去と誠実なフォールバック（依存なし・即日）

**目的**: API 導入前でも「測っていないものを測ったふりして表示する」状態を止める。

1. `simulatePronResult()` を削除し、暫定実装として **Web Speech API ベースの簡易評価** に置換する:
   - `SpeechRecognition` で認識テキストを取得し、リファレンステキストと単語アラインメント（レーベンシュタイン距離ベース）を取る
   - 一致した単語 = good、認識されたが不一致 = bad として単語ピルに反映（ok は使わない）
   - スコアは「認識一致率」1本のみ表示し、ラベルを「発音」ではなく **「認識一致率（簡易）」** に変更。流暢さ・イントネーション・完全性のバーは Phase 1 まで **非表示** にする
2. `renderProgress()` の音素ランダム初期化（`40+Math.random()*55`）を削除。実データがない音素は「未測定」表示にする
3. 会話終了スコアの乱数を削除し、Phase 2 までは「ターン数・発話語数」など**実測できる量のみ**表示する
4. `animateFakeWave()` / `animateFakeConvWave()` を削除。マイク不許可時は波形を止めて「マイクを許可してください」を表示する
5. 語彙スキルの固定値 80 更新を削除
6. UI に評価方式の明示を追加（フッター等）:「簡易モード: ブラウザ音声認識による一致率のみ。発音の質は評価していません」

**受け入れ基準**
- [ ] `grep -n "Math.random" index.html` の結果が UI 演出（該当なしが理想）のみになる
- [ ] マイクを拒否した状態で録音してもスコアが表示されない
- [ ] 無音で録音してもスコアが表示されず「聞き取れませんでした」となる
- [ ] 何をどう評価しているかの説明文が UI に存在する

### Phase 1: Azure Pronunciation Assessment 導入（発音ドリルを本物にする）

**目的**: 発音ドリルのスコアを全て実測値にする。

1. `api/speech-token.js` を新規作成:
   ```js
   // POST不要・GET。Azureの issueToken エンドポイントを叩き
   // { token, region } を返す。キャッシュ9分。
   export default async function handler(req, res) {
     const r = await fetch(
       `https://${process.env.AZURE_SPEECH_REGION}.api.cognitive.microsoft.com/sts/v1.0/issueToken`,
       { method: 'POST', headers: { 'Ocp-Apim-Subscription-Key': process.env.AZURE_SPEECH_KEY } }
     );
     res.status(200).json({ token: await r.text(), region: process.env.AZURE_SPEECH_REGION });
   }
   ```
2. `index.html` に Speech SDK を導入（CDN: `https://aka.ms/csspeech/jsbrowserpackageraw` を script-src に追加、または npm 化）
3. 録音フローを SDK に置換:
   ```js
   const paConfig = new SpeechSDK.PronunciationAssessmentConfig(
     referenceText,
     SpeechSDK.PronunciationAssessmentGradingSystem.HundredMark,
     SpeechSDK.PronunciationAssessmentGranularity.Phoneme,
     /*enableMiscue*/ true
   );
   paConfig.phonemeAlphabet = 'IPA';
   paConfig.enableProsodyAssessment = true;   // en-US のみ有効
   ```
4. 結果マッピング:
   - `PronScore` → `#r-score` / `AccuracyScore` → 発音 / `FluencyScore` → 流暢さ / `ProsodyScore` → イントネーション / `CompletenessScore` → 完全性
   - 単語ピル: `AccuracyScore >= 80` good / `>= 60` ok / それ未満 or `ErrorType !== 'None'` bad。`ErrorType` が Omission なら「読み飛ばし」、Insertion なら「余分な語」とツールチップ表示
   - `Words[].Phonemes[]` を `state.phonemeScores` に **EMA（α=0.3）で蓄積**（ランダム初期化は既に撤去済み）。進捗画面は蓄積済み音素のみ表示
5. `vercel.json` の CSP 更新:
   - `connect-src 'self' https://*.api.cognitive.microsoft.com wss://*.stt.speech.microsoft.com https://*.tts.speech.microsoft.com`
   - SDK を CDN で読む場合は `script-src` にも追加
6. **既知の落とし穴への対策（必須）**:
   - **無音・低品質音声で満点が返るケースが報告されている** → 録音の RMS 音量と発話時間（`Duration`）を検証し、しきい値未満（例: 発話 0.5 秒未満）なら「評価不能」を表示してスコアを破棄する
   - **ProsodyScore が 0 で返るバグ報告あり** → `enableProsodyAssessment` はプロパティ設定で行い、0 が返った場合はイントネーションバーを「—」表示にする（0点として平均に入れない）
   - ネットワーク断・API エラー時は**絶対にダミースコアにフォールバックしない**。エラートーストを出してリトライボタンを表示
7. 生レスポンス JSON を `state.audit[]` に直近20件保存（後述の検証プロトコルで使用）

**受け入れ基準**
- [ ] 正しく読んだ場合とわざと崩して読んだ場合（例: "Nice to meet you" を「ナイス ツー ミート ユー」とカタカナ読み）でスコアが明確に分離する（20点以上差）
- [ ] 単語を1つ飛ばすと該当単語が Omission として bad 表示される
- [ ] 無音録音で「評価不能」となる（満点にならない）
- [ ] オフライン時にスコアが一切表示されない
- [ ] APIキーがブラウザの DevTools（Network/Source）から見えない
- [ ] 音素グリッドが実測値のみで構成される

### Phase 2: 会話AIの本物化（Claude API + unscripted 評価）

**目的**: キーワードマッチ劇場を、実際の対話 + 実測フィードバックに置換する。

1. `api/chat.js` を新規作成。Claude API（`claude-sonnet-5`）を呼ぶ:
   - system prompt にトピック設定（`TOPICS` の設定を流用: ロール、シーン、ターン数目安、学習者レベル）
   - 「毎ターン: (1) 自然に返答 (2) 学習者の直前発話に文法・語彙の改善点があれば `correction` フィールドで1つだけ返す」を JSON 出力で強制
   - `max_tokens` 300 程度、会話履歴はクライアントから送る（ステートレス）
2. `TOPICS[].script` / キーワードマッチ処理を削除。`opener` と персона設定は system prompt の素材として残す
3. 会話中の各発話を Azure の **リファレンステキストなし評価**（unscripted）にかけ、ターンごとに発音・流暢さをバックグラウンド蓄積
4. セッションサマリーを実測値で再構成: 平均発音スコア / 平均流暢さ / 総発話語数 / correction 一覧（今日の学び）
5. TTS を Azure Neural TTS（例: `en-US-JennyNeural`）に置換。失敗時のみ speechSynthesis にフォールバック
6. `addVocabTip()` の固定辞書を削除（Claude の correction で代替）

**受け入れ基準**
- [ ] スクリプトにない話題（例: "I went scuba diving with my grandmother"）に文脈の通った返答が返る
- [ ] 文法ミスを含む発話（例: "I go to Tokyo yesterday"）に correction が表示される
- [ ] セッションサマリーの数値がすべて実測由来である
- [ ] ANTHROPIC_API_KEY がブラウザから見えない

### Phase 3: 進捗データの信頼性（測定の継続性）

1. スキルレベル（発音/流暢さ/イントネーション/語彙）を全て実測スコアの EMA に統一。語彙は unscripted 評価の VocabularyScore を使用
2. 音素別精度に「測定回数 n」を併記（n<3 は「データ不足」表示）— 1回の偶然で音素が赤/緑にならないようにする
3. スコア履歴に評価モード（`azure` / `simple`）をタグ付けし、混在平均しない（Phase 0 の簡易スコアと Azure スコアは別物）
4. localStorage スキーママイグレーション: 旧ランダムデータ（`phonemeScores` 等）は初回ロード時に破棄

**受け入れ基準**
- [ ] 新規ユーザーの進捗画面に 0 以外の数値が一切表示されない（全て「未測定」）
- [ ] 音素スコアに n が表示される

### Phase 4: 検証プロトコル（キャリブレーション）— 「信頼に足る」ことの証明

信頼性は実装しただけでは主張できない。以下を実施し、結果を `docs/検証結果.md` に記録すること。

1. **分離性テスト**: 同一フレーズ10種について
   - (a) ネイティブ音源（TTS で生成した音声をマイク経由で再生でも可）
   - (b) 日本語話者の丁寧な発話
   - (c) 意図的なカタカナ英語
   を各3回採点し、平均スコアが (a) > (b) > (c) の単調順序になることを確認。(a) と (c) の差が 15 点未満なら設定を見直す
2. **再現性テスト**: 同一録音（ファイル入力）を5回採点し、スコアの分散が ±3 点以内であることを確認
3. **既知エラー検出テスト**: 日本人学習者の典型誤り（r/l、th→s、語末子音への母音挿入 "cat"→"catto"）を意図的に発話し、該当音素・単語が低スコアになることを確認
4. **無音・雑音・非英語テスト**: 無音 / テレビ音声 / 日本語発話 で「評価不能」または明確な低スコアになること（満点バグの検出）
5. 上記の限界事項（雑音下では精度が落ちる、プロソディは en-US のみ、子供の声は精度低下等、Microsoft の Characteristics and limitations 記載事項）を UI の「評価について」モーダルに明記する

**受け入れ基準**
- [ ] `docs/検証結果.md` に4テストの実測ログ（生JSON参照付き）がある
- [ ] UI に評価エンジン名と限界事項の開示がある

## 6. 恒久ルール（このプロジェクトの掟）

1. **乱数でスコアを作らない。** 演出用の乱数（波形の見た目等）も、測定値と誤認され得る場所では使わない
2. **API エラー時にダミー値へフォールバックしない。** 「評価不能」はバグではなく誠実な仕様
3. **測っていない指標のバーを表示しない。** 表示する全数値は出所（API フィールド名）を説明できること
4. **APIキーは必ずサーバ側。** ブラウザに渡すのは短命トークンのみ
5. **評価方式をユーザーに開示する。** エンジン名・スコアの意味・既知の限界
6. スコア閾値（good/ok/bad の 80/60 等）を変える場合は Phase 4 の分離性テストを再実行する
