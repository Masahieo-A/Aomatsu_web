# 青松AI 全体要件定義書 v0.1

## 1. プロジェクト概要

### 1.1 目的
青松のように考え、書き、判断するAI「青松AI」を構築し、発信活動（Note/X執筆、コンテスト応募、提案書作成、企画立案）の壁打ち相手・下書き生成者・添削者として運用する。

### 1.2 スコープ
- **対象**: 本人専用。一般公開しない
- **対象外**: 音声・アバター再現、他者への応答代行、公開チャットボット化
- **成功の定義**: 青松AIの出力を本人が「7割方使える下書き」と感じる状態。および、ブラインドテストで身近な人の判別率が有意に下がること

### 1.3 基本思想: 3層モデル
「自分のようなAI」を以下の3層に分解し、層ごとに異なるデータと再現手法を割り当てる。

| 層 | 内容 | 再現手法 | 主データ源 |
|---|---|---|---|
| 文体層 | 語彙、リズム、引き算の美学、駄洒落 | few-shot例文 + スタイルガイド | 公開文章、添削差分 |
| 思考層 | 価値観、判断基準、論の立て方 | ペルソナカーネル + 判断記録RAG | 収集アプリ、AI対話ログ、添削差分 |
| 知識層 | 経歴、プロジェクト、人間関係、専門知識 | 事実DB（RAG） | 棚卸し文書、収集アプリ |

## 2. ユースケース定義

| ID | ユースケース | 入力 | 出力 | 優先度 |
|---|---|---|---|---|
| UC1 | 下書き生成 | テーマ・骨子・媒体指定 | 青松文体の下書き | 高 |
| UC2 | 青松らしさ添削 | 本人または他AIの草稿 | 修正案 + 修正理由（「青松ならこう直す」） | 高 |
| UC3 | 企画壁打ち | 企画案・アイデア | 青松視点の批評・懸念・発展案 | 高 |
| UC4 | 過去見解の検索 | トピック | 過去に自分が述べた立場・論拠の要約と出典 | 中 |
| UC5 | 反応シミュレーション | 発信予定の文章 | 「青松の読者/同僚/管理職ならどう受け取るか」の予測 | 低 |

## 3. データソース設計

### 3.1 ソース一覧とバイアス評価

**重要な前提**: 各データソースは青松の思考の異なる断面を、異なる偏りで切り取ったサンプルである。単一ソースへの過剰適合を避け、意識的に重み付けする。

| ソース | 量 | 捉えている断面 | 既知のバイアス | コーパス内の役割 |
|---|---|---|---|---|
| AI対話ログ（Claude） | 大 | 思考の掘り下げ・推敲過程・専門的探究 | 「AIに聞きたいこと」に特化。仕事・知的探究に偏り、日常判断・対人領域が欠落。Claudeの応答様式に合わせた質問の癖 | **補助**: 思考層の推論パターン抽出 |
| AI対話ログ（Gemini/GPT） | 大 | 同上（別モデル向けの聞き方） | 同上。ただしClaudeログと突き合わせることでモデル固有の適応バイアスを相殺できる | **補助 + バイアス検証**: 3モデル横断で共通する部分こそ「青松本来の思考」とみなす |
| 公開文章（Note/X、応募書類） | 中 | 推敲済みの完成形文体、公的な立場表明 | よそ行きの文体。本音や迷いは削られている | **正典（文体層）** |
| 業務文書（提案書、稟議、教材） | 中 | フォーマル文体、組織内での論の立て方 | 様式への適応。個人の声は薄い | 文体層のフォーマル面 |
| 収集アプリ回答 | 新規収集 | AIに聞かない領域を含むフラットな価値観・判断・美意識 | 自己申告バイアス（理想の自分を答えがち） | **正典（思考層）** |
| 添削差分 | 運用開始後に蓄積 | 実際の選好（語るのではなく直した跡） | なし（行動データのため最も信頼できる） | **最重要**: 全層の校正データ |

### 3.2 バイアス補正の運用原則
1. **正典の優先順位**: 添削差分 > 収集アプリ回答・公開文章 > AI対話ログ。矛盾したら上位を採用する
2. **3モデル交差検証**: Claude/Gemini/GPTのログから抽出した思考パターンは、2モデル以上で共通して観察されるものだけをペルソナカーネルに昇格させる。単一モデルにのみ現れるパターンは「そのモデルへの適応」と疑う
3. **欠落領域の明示**: AIログでカバーされない領域（日常判断、対人感情、美意識、雑談の呼吸）は収集アプリで能動的に埋める。カバレッジマップを維持する
4. **自己申告の検証**: アプリ回答（「こう答えたい自分」）と添削差分（「実際にこう直す自分」）が食い違ったら添削差分側を採用し、カーネルに注記する

### 3.3 データパイプライン

```
[エクスポート] → [前処理] → [抽出・タグ付け] → [コーパス統合] → [カーネル/RAG反映]
```

- **エクスポート**: Claude（設定→データエクスポート）、ChatGPT（conversations.json）、Gemini（Google Takeout）。公開文章はNote/XのAPIまたは手動収集
- **前処理**:
  - ユーザー発話と、ユーザーが承認・修正した最終成果物のみを抽出（AIの応答そのものは学習対象にしない。青松の声ではないため）
  - **匿名化（必須）**: 生徒氏名・成績・進路情報、同僚の実名、学校の機微情報を機械 + 目視の二段階で除去。生徒情報は仮名化ではなく削除を原則とする
- **抽出・タグ付け**: LLMで各断片に `{層, トピック, 文体レジスタ(公/私), 確信度, 出典}` を付与
- **統合**: 後述のコーパス4ファイル構成に集約。全断片に出典IDを保持し、後から遡れるようにする

## 4. アーキテクチャ

### 4.1 方針
ファインチューニングは行わない（データ量・更新頻度・コストの観点で不利）。**ペルソナカーネル（system prompt）+ 構造化コーパス（RAG）+ タスク別few-shot** の構成とする。

### 4.2 コーパス構成（4ファイル + DB）

| 成果物 | 内容 | 形式 |
|---|---|---|
| ①ペルソナカーネル | 価値観、判断原則、禁則、口癖、思考の癖。FABLE_LEGACY.mdを母体に拡張 | Markdown（~3,000字） |
| ②スタイルガイド | 文体規則を良例・悪例ペアで記述。媒体別（Note/X/公文書）のレジスタ切替 | Markdown |
| ③事実DB | 経歴、プロジェクト、人間関係、専門領域。時点情報付き | 構造化Markdown or Supabase |
| ④タスク別few-shot集 | UC1〜UC5ごとの入出力例（本人が採点済みのもの） | JSONL |

### 4.3 段階的実装

**Phase 1: Claude Project運用（構築1週間）**
- コーパス4ファイルをProjectナレッジに載せ、即日運用開始
- 目的: 最速で添削差分の収集ループを回し始めること
- 限界が見えたら（ナレッジ容量、検索精度、差分記録の手間）Phase 2へ

**Phase 2: 専用アプリ（構築2〜3週間）**
- スタック: Next.js + Supabase(pgvector) + Claude API + Vercel（既存の技術資産を流用）
- 追加機能: RAG検索、添削差分の自動記録（AI出力→本人編集→diff保存をUI内で完結）、出典表示付き回答（UC4）、モデル切替（Claude/Gemini比較）
- 収集アプリ（別紙要件定義）とSupabaseを共有し、回答データを直接参照

### 4.4 更新フロー
- 添削差分・アプリ回答は週次でコーパス候補に自動整形 → 本人が月次でレビューし採否決定 → カーネル/スタイルガイドをバージョン更新（v0.1, v0.2...）
- カーネルはGit管理し、変更履歴を残す（人格の変化も記録として面白い）

## 5. 評価計画

| 評価 | 方法 | 頻度 | 合格基準 |
|---|---|---|---|
| ブラインドテスト | 本人文章とAI文章を混ぜ、配偶者・同僚に判別してもらう | 月次 | 判別率が70%→55%へ低下 |
| タスク採点 | UC1〜UC3の出力を本人が5段階採点 + 修正率記録 | 常時 | 平均3.5以上、修正率50%以下 |
| 思考再現テスト | 過去に本人が実際に下した判断（記録済み）を隠してAIに問い、一致度を測る | 四半期 | 定性評価 |
| 欠落検出 | 「青松AIが答えられなかった/明らかに外した」質問をログ化 | 常時 | 次月の収集アプリ質問テーマに反映 |

## 6. セキュリティ・倫理

- 生徒・保護者の個人情報はいかなる形でもコーパスに含めない（前処理で削除徹底）
- 学校の未公開文書は事実DBに入れる場合も外部共有不可のフラグを付与
- 青松AIの出力を外部発信する際は必ず本人が最終確認する（AIの名義発信はしない）
- API課金上限をSupabase側で管理（月額上限アラート）
