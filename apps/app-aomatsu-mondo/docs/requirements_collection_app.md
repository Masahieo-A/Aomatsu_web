# 青松問答（情報収集アプリ）要件定義書 v0.1

## 1. 概要

### 1.1 目的
青松AIのコーパスのうち、既存データ（AI対話ログ・公開文章）ではカバーできない領域を**フラットにサンプリング**するための、本人専用の日次質問回答アプリ。

### 1.2 位置づけ
- 本アプリは独立プロダクトではなく、**青松AI構築プロジェクトのデータ収集フェーズを担う装置**である。開発時は全体要件定義書(requirements_aomatsu_ai.md)を必読とし、特に「3. データソース設計」「4.2 コーパス構成」との整合を保つこと
- 青松AI全体要件定義書の「収集アプリ回答（思考層の正典）」を生産する装置
- AI対話ログの偏り（仕事・知的探究への集中）を補正し、日常判断・対人・感情・美意識・雑談領域を能動的に埋める
- 本アプリの回答データはSupabaseに構造化保存され、青松AI本体（Phase 2）から直接参照される

### 1.3 設計原則
- **自由ペース**: 全質問(約350問)を最初から一覧公開し、回答順序・ペースは本人が決める。アプリ側でペース配分は行わない
- **途中保存は必須**: 書きかけの回答は自動保存され、いつでも中断・再開できること。回答が失われる事態は絶対に避ける
- **理由の言語化を強制**: 結論だけの回答はコーパス価値が低い。「なぜ」をセットで記録する
- **カバレッジの可視化**: 出題の強制はしないが、カテゴリ別の回答状況を常に見せ、手薄な領域(特にQ5: AIログ欠落域)へ本人が自律的に向かえるようにする

## 2. ユーザー・認証

- ユーザー: 青松本人のみ（シングルユーザー）
- 認証: Google OAuth（Supabase Auth）。許可メールアドレスをホワイトリスト固定
- デバイス: モバイルファースト（PWA対応が望ましい）

## 3. 質問バンク設計

### 3.1 カテゴリ定義

| ID | カテゴリ | 狙う層 | 例 | 初期問数 |
|---|---|---|---|---|
| Q1 | 価値観二択 | 思考層 | 「正確さと面白さ、片方削るならどっち？」 | 60 |
| Q2 | 判断エピソード | 思考層 | 「最近、頼まれごとを断った場面と理由は？」 | 50 |
| Q3 | 意見表明 | 思考層 | 「宿題は必要か。あなたの立場と論拠は？」 | 60 |
| Q4 | 文体A/B | 文体層 | 同内容の2文を提示「自分らしいのはどっち？何が違う？」 | 40 |
| Q5 | 日常・対人・感情 | 思考層（AIログ欠落域） | 「機嫌が悪い時、周囲にどう振る舞う？」「腹が立った直近の出来事は？」 | 60 |
| Q6 | 美意識・好み | 文体・思考層 | 「『いい文章』の条件を3つ挙げると？」「許せないデザインは？」 | 40 |
| Q7 | 事実・経歴 | 知識層 | 「教員を志した経緯は？」「陸上部で一番印象的な指導場面は？」 | 40 |

計 約350問を初期投入。

### 3.2 質問バンクの生成と監修
1. LLM（Claude API）でカテゴリ別に候補を一括生成（本要件定義と青松のプロフィール概要をプロンプトに含める）
2. 本人が管理画面で監修: 採用/修正/却下。**この監修作業自体もデータになる**（却下理由を任意入力→思考層データとして保存）
3. 運用開始後、青松AI本体の「欠落検出ログ」（答えられなかったテーマ）から追加質問を月次生成

### 3.3 文体A/B問題（Q4）の素材生成
- 本人の公開文章から1文を抽出し、LLMで「意味は同じだが文体が違う」対照文を生成してペア化
- 「本人の原文 vs 生成文」を混ぜ、どちらが原文か本人にも伏せる（自己認識と実際の文体のズレを検出できる）

## 4. 機能要件

### 4.1 質問一覧・回答導線（コア機能）
- 全質問（約350問）を最初から一覧表示。ページ配分やロック機能は設けない
- フィルタ: カテゴリ別 / 状態別（未回答・下書き・回答済み・スキップ）
- 並び順: カテゴリ順（デフォルト）/ ランダム / 下書きあり優先
- 一覧の各項目に状態バッジを表示し、「下書きあり」は一目で分かること
- 回答画面は前後の質問へスワイプ/ボタンで移動でき、一覧に戻らず連続回答できる
- 回答済み質問への再回答: Q1・Q3は6ヶ月経過後に「再回答」ボタンが出現し、旧回答を残したまま新回答を追記する（意見の時間変化もデータ）

### 4.2 途中保存（必須要件）
- 回答入力は**自動保存**する: 入力停止から2秒後（debounce）およびフォーカス喪失時・画面遷移時に下書きとしてSupabaseへ保存
- 回答は `draft`（下書き）と `submitted`（確定）の2状態を持つ。確定は明示的な送信ボタンで行う
- 下書き状態でアプリを閉じても、次回その質問を開いた時に入力内容が完全復元されること
- オフライン・通信失敗時はlocalStorageに退避し、復帰時に同期する（保存失敗をユーザーに気づかせるトースト表示も行う）
- 確定済み回答も後から編集可能（編集履歴として updated_at を更新）

### 4.3 回答UI
- 回答形式: カテゴリごとに最適化
  - Q1/Q4: 選択ボタン + 理由テキスト（理由は必須、最低20字）
  - Q2/Q3/Q5/Q6/Q7: 自由記述
- **音声入力**: Web Speech APIで音声→テキスト。話し言葉のまま保存するオプション付き（話し言葉の文体もデータになるため、整形せず残す選択肢を用意）
- スキップ可（スキップ理由を1タップ選択: 「答えたくない」「思いつかない」「質問が悪い」→ 質問バンク改善に利用）
- 回答確定後の任意深掘り: Claude APIが回答を読み「なぜ？」系のフォローアップを1問だけ生成。回答は任意

### 4.4 進捗ダッシュボード
- カテゴリ別回答数・カバレッジのヒートマップ（未回答が手薄な領域の可視化）
- 総回答文字数・回答済み/下書き/未回答の内訳（蓄積の可視化がモチベーション維持に効く）

### 4.5 管理画面
- 質問バンクのCRUD、LLM一括生成 → 監修フロー
- 回答の閲覧・編集・タグ修正
- エクスポート: JSONL（コーパス統合用）/ Markdown（人間レビュー用）

### 4.6 エクスポート仕様
青松AI本体のコーパス統合パイプラインが直接読める形式で出力する。

```jsonl
{"id": "ans_0123", "category": "Q2", "question": "...", "answer": "...",
 "reason": "...", "followup_q": "...", "followup_a": "...",
 "layer": "thinking", "topics": ["仕事", "断り方"], "register": "private",
 "input_mode": "voice_raw", "answered_at": "2026-07-20T07:45:00+09:00",
 "revision_of": null}
```

## 5. データモデル（Supabase）

```
questions:  id, category, body, body_options(jsonb, A/B用),
            source(seed/llm/gap_detection), status(draft/approved/rejected),
            reject_reason, reask_after(date, 再出題用), created_at
answers:    id, question_id, status(draft/submitted), answer_text, reason_text,
            choice, followup_q, followup_a,
            input_mode(text/voice_raw/voice_edited),
            skipped(bool), skip_reason, revision_of(自己参照, 再回答用),
            created_at, updated_at, submitted_at
coverage:   category, answered_count, draft_count, target_count(view で可)
```

- エクスポート対象は `status = submitted` のみ（下書きはコーパスに含めない）

## 6. 技術スタック

| 項目 | 選定 | 備考 |
|---|---|---|
| フロント | Next.js (App Router) + Tailwind | 既存資産と同構成 |
| バックエンド | Next.js API Routes | 規模的にFastAPI不要 |
| DB/認証 | Supabase (Postgres + Auth) | 青松AI本体と同一プロジェクトで共有 |
| LLM | Claude API | 質問生成・フォローアップ生成・タグ付け |
| 音声入力 | Web Speech API | 追加コストなし。精度不足ならWhisper APIを検討 |
| ホスティング | Vercel | |

## 7. 非機能要件

- 一覧から3タップ以内で回答入力を開始できる
- 保存の信頼性を最優先: 自動保存の失敗は必ずユーザーに通知し、入力内容をクライアント側に保持する
- LLM呼び出し失敗時も回答フローは続行可能（フォローアップはベストエフォート）
- データは本人のみアクセス可（RLS設定必須）
- Claude API月額上限の設定（想定: フォローアップ生成は1日5回 + 月次バッチのみで低コスト）
